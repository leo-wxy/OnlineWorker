OnlineWorker Rendering Strategy Preview

不是只选样式，而是选渲染方案

下面每个方案代表不同管线。左侧是数据怎么从 assistant reply 变成 UI，右侧是同一段内容的最终效果。 重点比较：安全性、Markdown 兼容度、是否可做富组件、落地复杂度、是否适合当前 Session Tab。

安全：能否避免 HTML 注入 兼容：GFM/表格/代码块 可控：组件和样式边界 维护：后续 bug 成本 效果：阅读和聊天体验

方案 1：Markdown AST → React 组件渲染

当前方向的正确版本。保留 react-markdown，修正 inline/block code 判断，为 h/p/li/code/pre/table/blockquote 定制组件和 CSS。 这是安全性、效果和维护成本最平衡的方案。

Markdown 渲染判断

TG 日志里的 最终回复已按 HTML edit 是 Telegram API 的格式化方式，不代表 App Session Tab 使用 HTML 注入。

App 侧仍应使用 react-markdown。关键是把 displayMode: "markdown" 的内容交给正确的 Markdown 组件渲染。

判断标准

  • 粗体 应该直接加粗，而不是显示 **粗体**。
  • 列表应该有正常缩进和项目符号。
  • inline code 是行内小胶囊，只有 fenced code 才是块。
如果粗体、列表、代码块都已经呈现为样式化内容，这轮验证可以判定通过。
function normalizeDisplayMode(role) {
  return role === "assistant" ? "markdown" : "plain";
}

方案 4：Markdown/HTML → sanitize → HTML 渲染

先把 Markdown 转为 HTML，允许部分 HTML 标签，再用 sanitizer 清洗，最后注入 DOM。视觉自由度高，但安全和边界成本最高。

HTML 渲染可以更自由，例如允许 mark 高亮、kbd、自定义 span class。

但这要求 sanitizer 精准控制标签、属性、URL scheme、style/class allowlist。否则 dangerouslySetInnerHTML 会变成长期风险点。

适用场景：内容来源可信、需要渲染非常复杂的 HTML 结构。当前 OnlineWorker assistant 回复不是这个场景。
