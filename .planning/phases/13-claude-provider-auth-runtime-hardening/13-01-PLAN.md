# 13-01 Plan: Harden Claude Provider Auth Runtime Readiness

**Status:** source-verified; installed-app verification pending explicit approval
**Created:** 2026-06-03
**Phase:** 13. Claude Provider Auth Runtime Hardening

## Problem

Claude provider sends can currently reach the user-visible streaming path before
Claude runtime/auth is known to be usable. When the native Claude CLI is logged
out, Telegram accepts the user message, OnlineWorker starts a provider turn, and
the actual `claude -p` subprocess fails later with:

```text
Not logged in · Please run /login
```

The previous runtime-env fallback is also fragile: it can scan active `claude`
processes with `pgrep` / `ps eww` and reuse `ANTHROPIC_*` values found there.
That is not a stable provider readiness model because it only works when a
matching process happens to be alive and readable.

## Goal

Make Claude provider readiness deterministic, explicit, and visible before user
traffic is accepted into a normal provider turn.

Claude should have one non-secret readiness contract that Dashboard, Telegram,
startup, and send-time preflight can all read. A logged-out Claude CLI, missing
CLI, stale runtime env, or missing configured runtime source should be surfaced
as provider-unavailable state, not as a late streaming error or apparent
no-response.

## Scope

In scope:

- Claude provider adapter/runtime readiness checks.
- Send-time preflight before spawning `claude -p`.
- Dashboard provider status payloads and rendering for Claude unavailable
  reasons.
- Telegram user-message handling for Claude provider-unavailable responses.
- Tests for CLI logged-in/logged-out, explicit runtime env, missing CLI, stale
  env, and no-active-process behavior.

Out of scope:

- Codex app-server approval behavior.
- Telegram topic route storage.
- Notification plugin routing.
- AI service configuration semantics from Phase 8.
- Storing raw Claude auth secrets in planning docs, logs, Dashboard payloads, or
  tests.

## Current Evidence

- Installed app received a Telegram message in Claude topic `7431`.
- `bot.handlers.message` routed the message to thread
  `5d90ac9d-2a08-45f6-8f3e-8b2bafb4edfa`.
- Claude turn started and Telegram message updates worked.
- Claude subprocess then failed with:

```text
Claude 执行失败 ... error=Not logged in · Please run /login
```

- Local real command returned:

```json
{"loggedIn": false, "authMethod": "none", "apiProvider": "firstParty"}
```

- Current process list had no active `claude` process, so process-env scanning
  had no usable source to reuse.
- Current OnlineWorker Claude provider config uses `bin: claude`.
- `ow-claude` is declared as an external wrapper capability, but it is not the
  explicit provider send auth/readiness contract for Telegram-managed sends.

## Design Direction

1. Introduce a Claude readiness model.

   Suggested shape:

   ```python
   {
       "ready": bool,
       "source": "cliAuth" | "runtimeEnv" | "missingCli" | "unknown",
       "reason": "ok" | "loggedOut" | "missingCli" | "staleEnv" | "missingRuntime",
       "authMethod": "subscription" | "oauth_token" | "proxyEnv" | "apiKeyEnv" | "authTokenEnv" | "none" | "",
       "checkedAt": "...",
       "detail": "non-secret human-readable reason",
   }
   ```

2. Make readiness the shared contract.

   Dashboard, `/status`, startup logs, Telegram send preflight, and provider
   runtime should all consume the same readiness result rather than duplicating
   auth checks.

3. Demote active-process env scanning.

   Active `claude` process env discovery must not be required for provider sends
   to work. Either remove it from the send path or keep it as diagnostic-only
   evidence that never determines normal readiness without an explicit config or
   CLI auth source.

4. Fail fast before `claude -p`.

   If readiness is false, provider send must not spawn the normal Claude
   subprocess. It should return a structured provider-unavailable result that
   existing Telegram/session event surfaces can render promptly.

5. Show the same reason in Dashboard and Telegram.

   The user should see a concise reason such as:

   ```text
   Claude provider unavailable: Claude CLI is not logged in.
   ```

   The message must not contain tokens, API keys, authorization headers, or raw
   env dumps.

## Implementation Tasks

1. Add Claude readiness data structure and checker.

   - Add a focused helper in `plugins/providers/builtin/claude/python/adapter.py`
     or a small adjacent module if it reduces adapter size.
   - Normalize `claude auth status` JSON and subprocess failures into the
     readiness shape.
   - Normalize explicit runtime env readiness into the same shape.
   - Preserve redaction for all secret-like values.

2. Rework `refresh_auth_status`.

   - Make it return/read from the readiness contract.
   - Keep existing `auth_ready` / `auth_method` compatibility properties for
     current status callers.
   - Ensure `loggedIn:false` becomes `ready=False`, `reason=loggedOut`.

3. Add send-time preflight.

   - Before `_send_user_message_once` spawns `claude -p`, check readiness.
   - If not ready, return a structured error without invoking Claude.
   - Ensure Telegram/session event rendering treats this as an immediate
     provider-unavailable response, not a hanging stream.

4. Update runtime/status surfaces.

   - Update Claude runtime `build_status_lines` and any Dashboard provider
     status path to include the readiness reason.
   - Ensure Dashboard can show Claude unavailable before a user sends traffic.
   - Keep wording short and non-secret.

5. Remove or demote process scanning.

   - Remove `_scan_active_claude_process_env` from normal readiness/send flow, or
     gate it behind a diagnostic-only path.
   - Add tests proving no active `claude` process is required and no process env
     scan is used as the main readiness source.

6. Add regression tests.

   Required cases:

   - Native CLI auth status logged in.
   - Native CLI auth status logged out.
   - Missing `claude` binary.
   - Explicit runtime env ready.
   - Stale localhost runtime env rejected.
   - No active Claude process does not change readiness.
   - Logged-out send path does not spawn `claude -p`.
   - Telegram/user-message path receives a prompt provider-unavailable result.
   - Dashboard/status payload exposes reason without secrets.

## Acceptance Criteria

1. `claude auth status` returning `loggedIn:false` is visible in Dashboard and
   `/status` before a user message is sent.
2. Sending a Telegram message to a Claude topic while Claude is unavailable gets
   a prompt provider-unavailable response.
3. Logged-out Claude does not spawn `claude -p`.
4. Active-process env scanning is not part of the normal provider send
   readiness path.
5. Runtime/env secrets are never logged or displayed.
6. Existing logged-in/native CLI and explicit runtime-env Claude flows still
   work.

## Risks

- Claude CLI auth output or command names can change, so parsing must degrade to
  a clear `unknown` / unavailable status instead of assuming ready.
- Explicit runtime env support must not accidentally reintroduce stale localhost
  proxy env or leak secrets.
- Dashboard status refresh must avoid expensive subprocess checks in tight loops.

## Completion Notes

Source implementation is complete as of 2026-06-03.

Implemented changes:

- Added a shared Claude readiness contract in the Claude adapter, including
  normalized `ready`, `source`, `reason`, `authMethod`, `checked_at`, and
  non-secret `detail` fields.
- Removed active-process environment discovery from normal Claude runtime env
  detection. Runtime env readiness now only comes from the current process
  environment, with stale localhost proxy values rejected.
- Changed Claude send preflight to force-refresh readiness before each send.
  Logged-out or unavailable Claude returns a structured provider-unavailable
  result and does not spawn the normal `claude -p` process.
- Added runtime-level preflight in the Claude provider send hook so Telegram
  sends fail before `resume_thread` and before provider message execution.
- Updated Telegram message runtime handling so provider `status:error` results
  become visible send failures instead of being logged as sent messages.
- Updated `/status` and provider owner bridge runtime status to surface Claude
  unavailable reasons as degraded state rather than blindly treating a connected
  adapter as healthy.
- Updated Dashboard/provider status tests so logged-out Claude is visible before
  user traffic.
