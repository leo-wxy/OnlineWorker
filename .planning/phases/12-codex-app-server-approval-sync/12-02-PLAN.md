# 12-02 Plan: Add Unix-Socket Shared Codex App-Server Transport

## Objective

Add first-class `unix://` app-server transport support for OnlineWorker-managed Codex sessions so OnlineWorker/TG and a visible Codex CLI can attach to the same local app-server without using a fixed TCP port.

The product validation entry point is the fixed OnlineWorker remote proxy socket:

```bash
alias codexR='/opt/homebrew/bin/codex --remote "unix:///Users/wxy/Library/Application Support/OnlineWorker/codex_remote_proxy.sock" --cd "$(pwd)"'
```

This is intentionally different from bare `--remote unix://`. Bare `unix://` connects to Codex's default app-server control socket and bypasses OnlineWorker's proxy, so it is useful for direct Codex transport validation but not for validating Telegram approval mirroring/control.

The target shared test shape is:

```bash
codex app-server --listen unix://PATH
codex --remote unix://PATH --cd "$(pwd)"
```

OnlineWorker should connect to or own the same app-server, expose a fixed local proxy socket to the visible CLI, mirror mapped approval requests to Telegram without hiding the Codex CLI native approval prompt, and relay Telegram decisions back to app-server when Telegram handles the request first.

## Decision

Use `unix://` as the primary local shared app-server transport for the next visible CLI + Telegram validation.

Rationale:

- `unix://` avoids fixed-port conflicts such as `4722` already being occupied.
- The socket is local filesystem state, so access can be limited with directory and socket permissions.
- It matches Codex app-server's documented supported local listener modes while avoiding a TCP listener.
- It can support the same shared-host topology as `ws`: one app-server, one approval request id, multiple local clients.
- Current Codex CLI 0.135.0 can start the default `unix://` listener with a temporary `CODEX_HOME`.
- OnlineWorker `AppServerProcess(protocol="unix")` can start that listener and `CodexAdapter.connect("unix://")` can initialize and call `model/list`.
- External arbitrary absolute app-server socket paths can still fail with `Operation not permitted (os error 1)`; custom app-server paths verified so far should stay under the Codex app-server control directory.
- The fixed OnlineWorker proxy socket lives under `~/Library/Application Support/OnlineWorker/`, has `0600` socket permissions with a `0700` parent directory, and is the supported visible CLI entry point for OnlineWorker approval testing.

Out of scope:

- unrelated Mac app UI changes; the Codex external CLI copy command is in scope for the fixed Unix proxy validation path
- packaging, install, or restart without explicit approval
- Remote Connection work
- non-Codex providers
- shared `core/` approval abstractions
- replacing native approval behavior for Codex Desktop, VS Code, or ordinary CLI sessions not attached to the OnlineWorker-managed app-server

## Implementation Steps

1. [x] Extend config normalization for `unix`.
   - Accept `transport.type: unix`.
   - Infer `unix` from `app_server_url` values beginning with `unix://`.
   - Add `shared_unix` as a valid `live_transport`.
   - Preserve existing default behavior: Codex app mode may continue using `stdio` unless the user config explicitly selects a shared transport.

2. [x] Add unix app-server process startup.
   - Generate `codex app-server --disable hooks --listen unix://` or `unix://PATH`.
   - Use `app_server_url` as the canonical endpoint when provided.
   - For custom socket paths, ensure the parent directory exists and handle stale socket files safely.
   - Return the exact `unix://...` endpoint to the runtime.

3. [x] Add adapter support for WebSocket over Unix domain socket.
   - Route `unix://` endpoints through a Unix-domain WebSocket client path.
   - Keep the app-server protocol unchanged: one JSON-RPC message per WebSocket text frame.
   - Keep `stdio://` JSONL handling unchanged.
   - Keep `ws://` behavior unchanged as fallback.

4. [x] Update runtime resolution and health behavior.
   - Treat external `app_server_url: unix://...` as a valid connection target.
   - Avoid HTTP `/readyz` probing for unix endpoints unless a real unix health probe exists.
   - Reuse existing connect/retry behavior for startup readiness.

5. [x] Update shared live transport helpers.
   - Add `shared_unix` alongside `shared_ws`.
   - Allow managed TUI/CLI host paths to receive `unix://...` as the remote target.
   - Keep remote-proxy text rewriting limited to transports it can correctly proxy; if the current proxy remains WebSocket-only, document and test that limitation instead of silently using it for unix.

6. [x] Add fixed Unix remote proxy behavior.
   - Default shared Unix app-server still uses Codex's `unix://` app-server control socket.
   - OnlineWorker exposes a separate fixed Unix proxy socket at `~/Library/Application Support/OnlineWorker/codex_remote_proxy.sock`.
   - Visible CLI command uses the OnlineWorker proxy socket, not bare `unix://`.
   - The proxy resolves client cwd from the Unix peer pid and injects it into `thread/list` requests, so `/resume` lists the current cwd's sessions instead of all history.
   - The proxy mirrors mapped approval requests to TG while forwarding the original request to CLI, preserving Codex native approval prompts.
   - The proxy forwards `serverRequest/resolved` to CLI and clears TG mirror pending state.
   - The proxy consumes expected relay close/reset/cancel exceptions so app restart or CLI disconnect does not leave unhandled task exceptions.

7. [x] Add focused automated tests.
   - Config accepts explicit `unix` transport and `unix://` app_server_url.
   - Process startup builds the expected `--listen unix://...` command.
   - Adapter selects unix transport for `unix://...` endpoints.
   - Shared-live helpers recognize `shared_unix`.
   - Remote proxy Unix socket relay, cwd-scoped thread list, approval mirror, duplicate approval dedupe, closed-upstream cleanup, and `serverRequest/resolved` cleanup are covered.
   - Existing `stdio` and `ws` tests continue passing.

8. [x] Validate installed fixed Unix remote proxy startup and packaging.
   - Build through the combined wrapper.
   - Install the generated DMG to `/Applications/OnlineWorker.app`.
   - Verify installed hashes and runtime socket ownership.
   - Verify app restart recreates `codex_remote_proxy.sock`, app-server reconnects, and `codex-owner-bridge` starts.

9. [ ] Validate the real fixed-session shared CLI + TG authorization case with Unix socket transport.
   - Start or restart installed OnlineWorker.
   - Attach a visible CLI with `codexR resume <session_id>`; do not use bare `--remote unix://` for this validation.
   - Trigger an approval from the visible CLI.
   - Verify Telegram receives a mirrored clickable approval for the same request id while the CLI native prompt remains visible.
   - Verify Telegram allow/deny resolves the Codex app-server request and the CLI converges.
   - If an explicit path is needed, prefer `unix://$CODEX_HOME/app-server-control/<name>.sock` over an arbitrary external path.

## Acceptance Criteria

1. `transport.type: unix` and `app_server_url: unix://...` are parsed as Codex unix app-server transport.
2. OnlineWorker can start Codex app-server with `--listen unix://` or `--listen unix://PATH`.
3. OnlineWorker can connect to a Codex app-server over Unix socket using the app-server WebSocket protocol.
4. `shared_unix` is recognized as a local shared live transport for managed Codex sessions.
5. `stdio` remains the private OnlineWorker-owned app-server path and is not described as externally shareable.
6. `ws://127.0.0.1:<port>` remains supported as a fallback shared transport.
7. Telegram approval controls are still shown only when OnlineWorker owns the active app-server request id and reliable thread/topic mapping.
8. Hook/current-session mirror paths remain notification-only by default.
9. Focused config/Codex tests and `git diff --check` pass.
10. Installed OnlineWorker exposes `codex_remote_proxy.sock` as the supported visible CLI entry point for Telegram approval mirroring/control.
11. Arbitrary external `unix://PATH` values that fail with local Codex CLI `EPERM` are documented as path/runtime limited, not as a general Unix transport blocker.
12. Real fixed-session Unix proxy + visible CLI + Telegram approval validation passes before using Phase 12 for product validation.
