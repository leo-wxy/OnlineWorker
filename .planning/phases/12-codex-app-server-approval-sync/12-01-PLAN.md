# 12-01 Plan: Build OnlineWorker-Managed Codex App-Server Approval Host

## Objective

Follow the reference host/client model used by Paseo, Happy, and the Codex IDE extension. For OnlineWorker-managed Codex sessions, OnlineWorker owns the Codex app-server request/response channel, renders Telegram as the remote approval UI, and relays Telegram decisions back to the same app-server request id. Existing Codex Desktop, VS Code, and ordinary CLI sessions keep native approval behavior and are mirror-only unless a future controlled proxy path explicitly makes OnlineWorker the host.

Reference projects and model:

- `getpaseo/paseo`: Provider-owned Codex app-server client registers `item/commandExecution/requestApproval`, `item/fileChange/requestApproval`, and `item/tool/requestUserInput` handlers, stores pending permission state, and responds to the same request id.
- `slopus/happy`: `happy codex` starts `codex app-server --listen stdio://`, maps app-server approvals into Happy's permission handler/mobile UI, and writes the decision back to app-server.
- Codex IDE extension: official product analogy for a Codex host/client. It uses Codex CLI and shared `~/.codex/config.toml`; useful for the "OnlineWorker can be a host" model, not for attaching to an already running Desktop/IDE session.

## Non-Goals

- No Remote Connection work.
- No readiness probe for Remote Connection.
- No unix-socket shared app-server transport in this plan; that is tracked separately in 12-02.
- No Mac app UI changes.
- No shared `core/` approval abstraction in the first pass.
- No Telegram/other IM surface redesign in this phase.
- No suppression or replacement of native approval prompts for Codex Desktop, VS Code, or ordinary CLI sessions that OnlineWorker does not own.
- No claim that OnlineWorker can attach to an already running Codex Desktop/VS Code approval stream.
- No clickable Telegram approval controls for hook/current-session mirror-only paths by default.
- No packaging, install, or restart without explicit approval.

   - app-server server request callbacks
   - `item/commandExecution/requestApproval`
   - `item/fileChange/requestApproval`
   - `tool/requestUserInput`

2. [x] Compare the reference host/client projects and define the Codex plugin ownership rule.
   - OnlineWorker-managed Codex sessions may show clickable Telegram approval controls because OnlineWorker owns the app-server request id and active adapter connection.
   - Codex Desktop, VS Code, and ordinary CLI sessions remain native approval clients. OnlineWorker mirrors them only unless it owns a controlled proxy/host path.
   - Telegram decisions are relayed back to app-server and resolved through the app-server lifecycle.
   - If app-server resolves first, Telegram pending state is cleared, disabled, or marked stale.
   - If Telegram resolves first, app-server receives the response and Codex-side state converges through the normal app-server resolution flow.
   - Ambiguous, duplicate, unmapped, hook-mirror, or current-session-mirror requests must not create global-topic Telegram approval controls.

3. [x] Add focused tests before behavior changes.

   - duplicate request keys do not produce duplicate pending Telegram approvals.
   - app-server-resolved request clears, disables, or marks Telegram state stale.
   - Telegram decision is relayed to app-server rather than completed locally.

   - Ensure request id used for app-server TG actions is the app-server JSON-RPC request id, not item id, thread id, or Telegram message id.

## Acceptance Criteria

1. OnlineWorker-managed Codex sessions follow the Paseo/Happy host/client pattern: OnlineWorker owns the app-server request/response channel.
2. Telegram receives a clickable approval prompt for app-server requests only when they are owned by OnlineWorker's active Codex adapter and mapped to a reliable thread/topic; non-app-server controls require an explicit controlled-host path or opt-in blocking hook.
3. App-server Telegram decisions are relayed to app-server as a response to the same JSON-RPC request id, not handled as an independent controller.
4. If app-server resolves first, Telegram pending state is cleared, disabled, or marked stale.
5. If Telegram resolves first, app-server receives the decision and Codex-side state converges through the normal app-server lifecycle.
6. Existing Codex Desktop, VS Code, and ordinary CLI native approval prompts are not intercepted, replaced, or controlled by OnlineWorker.
7. Hook/current-session mirror paths are notification-only by default and do not create clickable approval controls unless an explicit controlled-host path or opt-in blocking hook exists.
8. Missing thread/topic mapping never creates clickable approval controls in a provider global topic.
9. Duplicate or stale app-server requests do not create multiple active approvals.
10. Production implementation changes stay under `plugins/providers/builtin/codex/`.
11. Local focused Codex tests and `git diff --check` pass after the boundary alignment.
12. Real OnlineWorker-managed app-server/TG authorization chain is validated for the managed app-server path before calling the 12-01 host boundary done.
