# 04-01: Align Claude Existing-Session Resume Ownership

## Goal

Make Claude sends consistent across TG and the desktop Session Browser:
- Existing Claude sessions can be resumed and written.
- Imported/history sessions are not silently remapped into new sessions.
- Externally busy terminal sessions are not stolen by OnlineWorker.
- Concurrent sends to one Claude session are serialized.

## Implementation Steps

1. Update Claude runtime tests to require direct resume of imported/history sessions and rejection of externally busy sessions.
2. Remove the Claude imported-thread auto-remap path from provider message preparation.
3. Add a per-session send lock in the Claude adapter so competing TG/App sends cannot launch concurrent `claude --resume` processes for the same session.
4. Route desktop Claude Session Browser sends through the provider owner bridge by using the generic provider-session send command.
5. Keep explicit fork as future/explicit UX only; do not fork on normal message send.

## Verification

- Python tests for Claude runtime, adapter, TG handler, and provider owner bridge.
- Frontend/session browser API tests where available.
- Rust provider session command tests if send routing changes touch command behavior.
