# 06-01: Add Minimal Notification Channel Abstraction

## Goal

Introduce a small notification plugin mechanism so notification delivery is not limited to Telegram-only channels, while preserving existing Telegram task, approval, topic, streaming, and final-reply behavior.

## Scope

- Add a minimal notification event shape with:
  - `status`
  - `agent_name`
  - `task_name`
  - `message`
  - `task_id`
  - `agent_id`
- Support only three notification statuses:
  - `needs_action`
  - `failed`
  - `completed`
- Render notifications as:
  ```text
  {状态} · {Agent} · {任务}
  {一句话}
  ```
- Deduplicate notifications with:
  ```text
  dedupe_key = task_id + agent_id + status
  ttl = 24h
  ```
- Add a minimal notification router:
  ```text
  receive event -> dedupe -> send to enabled notification plugins
  ```
- Treat Telegram as one builtin notification plugin.
- Leave room for external/custom notification plugins such as WeChat.
- Keep ordinary progress, streaming deltas, and child-agent start/complete events out of notification delivery by default.
- Do not migrate existing Telegram business send paths in this slice.

## Design Rules

1. Notifications are concise status events, not replacements for existing Telegram task messages.
2. Notify only when the caller emits one of the supported statuses:
   - `needs_action`: approval/question/manual decision
   - `failed`: failed or blocked task/agent
   - `completed`: root task completed or all agents completed
3. Do not notify every progress event.
4. Do not notify every child-agent completion.
5. A repeated `{task_id, agent_id, status}` within 24 hours must not produce duplicate notifications.
6. Every notification must include both an agent label and a task label.
7. Core notification code must not know how each app sends messages; each notification plugin owns its own send logic.

1. [x] Add `core/notifications/` for event shape, router, registry, and in-memory dedupe.
2. [x] Add `plugins/notifications/builtin/telegram/` as the first notification plugin.
3. [x] Support external notification plugin discovery so a custom plugin such as WeChat can be inserted later without changing core notification routing.
4. [x] Add a first-class `Setup → Notifications` UI for enabling/disabling notification channels.
5. [x] Leave existing Telegram task, approval, topic, streaming, and final reply behavior unchanged.
6. [x] Add focused unit tests for:
   - notification formatting
   - dedupe key suppression
   - status changes producing separate notifications
   - router dispatching only to enabled plugins
   - Telegram plugin preserving existing send behavior

## Out of Scope

- Implementing non-Telegram channels in this repository.
- Migrating existing Telegram business messages to the notification router.
- Streaming delta notifications.
- Detailed task cards or notification center.
- Full replacement of Telegram as an input channel.
