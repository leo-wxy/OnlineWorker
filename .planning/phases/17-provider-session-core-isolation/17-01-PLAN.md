# 17-01 Plan: Isolate Provider Session Logic From Core

**Status:** in progress
**Created:** 2026-06-10
**Phase:** 17. Provider Session Core Isolation

## Problem

Provider-private session and workspace rules still leak into core desktop code.
The current Claude workspace mismatch exposed this directly: Telegram uses the
Claude Python provider facts path, while the Session Tab uses Rust code in
`mac-app/src-tauri/src/commands/claude.rs` to read `~/.claude/projects`
directly and duplicate Claude-specific filtering.

That duplication lets the same provider-private rule drift between App,
Telegram, Dashboard, TaskBoard, and future surfaces.

## Goal

Make provider session/workspace behavior fully provider-owned. Core/Tauri should
call generic provider session/facts commands and must not parse provider-private
stores, fields, hook payloads, or launch-specific metadata such as Claude
`entrypoint`.

## Scope

In scope:

- Inventory provider-specific logic currently living under Tauri core commands.
- Move Claude session/workspace list, read, send, archive, and recent-activity
  rules behind provider/plugin-owned capabilities or a single provider facts
  bridge.
- Remove duplicated Claude workspace/session filtering between Python provider
  and Rust Session Tab code.
- Replace `claude_projects`-style core special cases with generic provider
  session access where possible.
- Keep App Session Tab, TaskBoard, Dashboard recent activity, and Telegram
  behavior consistent through one provider-owned source of truth.
- Add parity tests proving App and Telegram see the same provider workspace set
  for meaningful manual CLI sessions, managed sessions, smoke sessions, login
  failures, and provider-private noise.

Out of scope:

- Rewriting the message bus event contract.
- Changing Phase 12 Codex approval ownership semantics.
- Changing Phase 16 external ingress ownership.
- Adding new provider product surfaces.
- Removing provider-specific code from provider plugins.

## Reference Anchors

- `plugins/providers/builtin/claude/python/storage_runtime.py`
- `plugins/providers/builtin/claude/python/provider.py`
- `mac-app/src-tauri/src/commands/claude.rs`
- `mac-app/src-tauri/src/commands/provider_sessions.rs`
- `mac-app/src-tauri/src/commands/dashboard/recent_activity.rs`
- `bot/handlers/workspace.py`
- `core/providers/facts.py`

## Planned Work Items

1. Map current provider-specific core leaks.

   - Claude Rust session list/read/send/archive commands.
   - Dashboard recent activity imports of Claude-specific skip logic.
   - Provider session list access modes that encode provider-private storage
     knowledge in core.

2. Define the provider-owned session/facts boundary.

   - Decide which commands must call the running owner bridge.
   - Decide which read-only paths can call packaged sidecar/provider facts.
   - Define failure and timeout behavior so the App remains responsive.

3. Migrate Claude session/workspace reads.

   - Keep Claude project parsing in the Claude provider/plugin layer.
   - Preserve meaningful manual Claude CLI sessions.
   - Preserve filtering for smoke/login/no-assistant noise.
   - Preserve initial history hydration and live update behavior.

4. Remove duplicated core filtering.

   - Delete or shrink Rust provider-private parsing helpers once call sites use
     provider-owned data.
   - Keep Tauri commands generic and provider-neutral.

5. Keep preview summaries provider-owned, incremental, and safe.

   - Session list rows may use provider-owned cached facts plus live activity
     merge, but must not require whole-history reload on every tab revisit.
   - Provider-neutral summary surfaces may keep an in-app snapshot cache of the
     last non-empty provider session rows so tab switches do not blank cards or
     force a full list reload before rendering.
   - Explicit user reload remains the boundary for forced full provider list
     refresh; normal tab revisit should reuse cached summary rows first and then
     merge incremental updates.
   - When provider session list rows miss a usable preview, the provider-owned
     boundary may hydrate a small number of active rows from provider history as
     a fallback summary instead of pushing that logic into React pages.
   - Preview fallback text shown in provider-neutral summary surfaces such as
     Session list cards and TaskBoard cards must be sanitized for sensitive
     names and absolute local file paths before display.
- Incoming empty/low-signal summary rows must not overwrite an already known
  richer preview in the in-app summary cache.
- Full session detail remains the provider read surface; summary-card
  sanitization must not mutate underlying session history.
- Hidden App tabs must not prewarm provider session-list caches with stale
  owner-bridge rows before the visible surface needs them. The first visible
  Session/TaskBoard load should be allowed to force-refresh provider session
  rows while still reusing local summary cache for immediate paint.

## Implementation Notes

- Session list cards and TaskBoard cards now prefer live summary fields over
  stale cached preview text, so event-bus updates can replace older summary
  rows instead of being hidden behind them.
- In-app summary cache merging now keeps richer cached previews when incoming
  rows are empty or low-signal, which avoids blank cards during tab revisit.
- Explicit reload remains the full-refresh boundary; normal tab revisit keeps
  cached summary rows first and then merges incremental updates.
- Focused regression coverage now passes for the session preview merge path,
  TaskBoard live preview precedence, and provider session row contract.
- Installed-app evidence on 2026-06-18 narrowed the remaining bug to hidden-tab
  preloading plus owner-bridge list caching: `list_sessions(force_refresh=true)`
  already returns the live Codex session and latest preview, while
  `session_activities` may still be stale. The visible `Session` and `TaskBoard`
  surfaces therefore need a first-visible provider list refresh instead of
  trusting a startup-time hidden preload.
- Installed-app evidence later on 2026-06-18 narrowed the remaining `TaskBoard`
  preview gap further: the live Codex row already exposes a non-empty provider
  preview through `list_sessions(force_refresh=true)`, but the TaskBoard card
  can still render blank because two frontend-only issues combine:
  `TaskBoard` first-visible provider refresh was scheduled through a separate
  effect and could be skipped while the initial refresh was still in flight,
  and active-session preview selection preferred dashboard
  `highlightedThreadPreview` even when that value only repeated the title while
  the provider row already had a richer preview. The minimal fix stays in the
  provider-neutral summary surface: serialize the first visible provider
  refresh after the initial refresh completes, then fall back from repeated
  dashboard preview text to the provider session preview.
- Installed-app evidence on 2026-06-20 narrowed the next remaining preview lag
  further: even when the provider row already has a newer `preview` from
  `list_sessions(force_refresh=true)`, a stale `session_activities`
  `lastAssistantMessage` can still win the running-card preview because the
  TaskBoard activity lane used activity preview unconditionally when present.
  The minimal follow-up stays local to TaskBoard model building: if the mapped
  provider session row is newer than the activity row and has a stronger
  provider-owned preview, prefer the provider preview for the running card
  instead of the stale activity text.
- Installed-app evidence later on 2026-06-20 narrowed the orphaned
  `CODEMAKER / OK` card to a separate frontend-only fallback bug. The running
  owner-bridge `session_activities` snapshot no longer contains a live
  Codemaker `OK` activity; it only contains one stale completed Codex activity.
  At the same time, `list_sessions(force_refresh=true)` reports the old
  Codemaker `OK` row with `providerActive=false`. The remaining false-running
  card therefore comes from TaskBoard trusting stale dashboard
  `recentActivity.activeSessionId/activeWorkspacePath` as a strong live signal
  even after the matched provider row explicitly says the session is inactive.
  The minimal fix stays in the provider-neutral summary surface: dashboard
  active-session fallback is allowed only when the dashboard summary itself is
  high-signal and the matched provider session row does not explicitly carry
  `providerActive=false`.
- Installed-app evidence on 2026-06-20 narrowed the next `SessionBrowser`
  issue to first-visible list hydration instead of provider preview generation:
  the `任务` page already shows the live Codex card with provider-owned preview,
  but the first switch into `会话` can still render `WORKSPACES 0 / 未找到会话`
  until the user clicks `Refresh`. The strongest explanation matched source
  behavior in the provider-neutral summary cache layer: a first-visible
  `forceRefresh` can momentarily return an empty provider session list, and the
  previous frontend merge path treated that empty result as authoritative,
  clearing cached session rows before a later refresh repopulated them. The
  minimal fix stays in the summary surface: first-visible auto refresh keeps
  the last known provider rows when the forced snapshot is transiently empty,
  schedules one bounded retry, and still preserves explicit user `Refresh` as
  the only path allowed to accept an authoritative empty snapshot.
- Fast packaged verification on 2026-06-20 closed that SessionBrowser branch:
  the installed app now enters `会话` with `WORKSPACES 25` plus loading
  placeholders instead of `WORKSPACES 0 / 未找到会话`, and then hydrates the
  real workspace/session rows without requiring a manual `Refresh`.
- The same packaged run on 2026-06-20 also confirmed the stale TaskBoard
  running-card fallback is gone in the installed app: only the two real
  `CODEX` running cards remain, both with preview text, and the orphaned
  `CODEMAKER / OK` card no longer appears after refresh.
- Summary-surface preview sanitization now also applies to TaskBoard model
  preview text so absolute local file paths do not leak when a live provider
  session preview is promoted ahead of stale activity projection.
- The same provider-owned runtime boundary now also owns Codex capacity
  recovery for managed sessions: when app-server reports `turn/completed` with
  the known capacity-abort reason, the Codex runtime may issue one guarded
  `thread/resume` plus `turn/start("继续")` follow-up instead of pushing this
  retry behavior into frontend surfaces.
- Installed-app verification on 2026-06-20 then narrowed the remaining
  session-detail flicker to a right-pane behavior issue rather than another
  list/data-source bug: `TaskBoard` and `SessionBrowser` rows were already
  correct, but `GenericProviderChat` still ran repeated
  `scrollIntoView({ behavior: "smooth" })` animations for passive message
  hydration coming from `loadMessages`, silent refresh, event-stream updates,
  and active-session polling. The minimal fix stays local to the detail pane:
  passive snapshot application now uses `auto` scrolling, while only
  user-authored optimistic appends keep `smooth`, so tab switches no longer
  replay several visible scroll animations.
- Fast packaged verification on 2026-06-20 completed again after this detail
  pane change, producing
  `OnlineWorker/mac-app/src-tauri/target/release/bundle/dmg/OnlineWorker_1.6.1_aarch64.dmg`
  and reinstalling the app to `/Applications/OnlineWorker.app`. This keeps the
  phase outcome grounded in installed-app evidence rather than source-mode
  behavior alone.
- Focused source follow-up now also closes one remaining provider-owned
  Dashboard gap: `recent_activity` no longer requires a workspace to have been
  pre-registered in `onlineworker_state.json` before App surfaces can show it.
  When provider session rows already expose a live workspace/session pair,
  Dashboard derives a synthetic `{tool}:{path}` workspace identity plus name
  directly from provider-owned rows and only falls back to state-only recent
  activity when provider rows are entirely absent. This keeps manual CLI
  workspaces closer to the same provider-owned truth used by Telegram
  `/workspace` and Session Tab, without reintroducing provider-private file
  parsing into Tauri core.

5. Verify parity.

   - Source tests for provider facts and Tauri provider sessions.
   - Regression tests for duplicate basename workspace display.
   - Installed-app validation that App Session Tab and Telegram `/workspace`
   report the same Claude workspace set.
   - Regression tests for active-session preview fallback, cached list reuse,
     and summary-surface sanitization.
   - Regression tests for running-card freshness arbitration when stale
     activity preview and fresher provider preview coexist for the same
     session.
   - Regression tests for first-visible SessionBrowser auto refresh so a
     transient empty provider list cannot blank the workspace/session columns
     while explicit user refresh still keeps full-refresh semantics.

## Success Criteria

1. Core/Tauri no longer reads Claude project jsonl files directly for workspace
   or session list behavior.
2. Claude-specific fields such as `entrypoint` are interpreted only by the
   Claude provider/plugin layer.
3. App Session Tab, TaskBoard/Dashboard recent activity, and Telegram workspace
   listing share one provider-owned session/workspace truth source.
4. Meaningful manual Claude CLI workspaces appear consistently in both App and
   Telegram.
5. Smoke, login-failed, no-assistant CLI noise, and provider-private noise are
   filtered consistently.
6. Focused source tests and installed-app verification pass before closure.
7. Provider-neutral summary surfaces do not expose raw sensitive names or
   absolute local file paths when preview fallback is hydrated from session
   history.
