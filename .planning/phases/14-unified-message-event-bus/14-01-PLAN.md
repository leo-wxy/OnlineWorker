# 14-01 Plan: Establish Unified Message Event Bus And First Projections

**Status:** source verified
**Created:** 2026-06-04
**Phase:** 14. Unified Message Event Bus

## Problem

OnlineWorker currently has multiple message-adjacent paths that each derive
their own state:

- Telegram sends and streaming edits route through `bot/events.py` and handler
  logic.
- App session sending uses Tauri commands and owner/provider bridge calls.
- TaskBoard currently derives cards from session lists and dashboard recent
  activity.
- Notification completion summaries derive from turn/final-reply handling.
- Approval and question flows have their own pending maps and Telegram UI state.

These paths work individually, but they make system behavior hard to explain and
easy to regress. A user-facing message can appear differently across App,
Telegram, TaskBoard, and notifications because those surfaces are not consuming
one shared message lifecycle.

## Goal

Add a single OnlineWorker message/event bus and a first session activity
projection.

All message and lifecycle entry points should publish normalized events. Existing
delivery behavior can remain in place, but new TaskBoard, notification, App, and
Telegram message behavior must have one shared event source.

## Scope

In scope:

- Core Python message event contracts.
- In-process publish/subscribe bus.
- Deterministic event id and dedupe key helpers.
- Initial session activity projection.
- Publishing from:
  - user message gateway results
  - Telegram send path
  - App/provider owner bridge send path
  - provider session event normalization
  - approval/question request and answer paths
  - final reply and turn completion paths
  - notification requested/emitted paths
- TaskBoard model changes that read the activity projection.
- Tests for bus ordering, dedupe, projection state, consumer failure isolation,
  and notification events.

Out of scope:

- Replacing Telegram polling.
- Replacing provider adapter protocols.
- Changing Codex app-server approval source-of-truth semantics.
- Adding external durable broker infrastructure.
- Changing notification plugin settings UI.
- Packaged-app verification unless explicitly approved in the current
  conversation.

## Design

### Core Contracts

Add a core module, likely under:

```text
core/messages/
```

Suggested files:

```text
core/messages/events.py
core/messages/bus.py
core/messages/projections.py
```

Initial event shape:

```python
@dataclass(frozen=True)
class MessageEvent:
    event_id: str
    kind: str
    provider_id: str
    workspace_id: str
    workspace_path: str
    session_id: str
    turn_id: str
    source: str
    created_at: float
    dedupe_key: str
    payload: dict[str, Any]
```

Event ids must be deterministic when the upstream event has a stable request id,
turn id, item id, or message id. Generated ids are acceptable for local-only
events but must not break dedupe.

Payloads must be public/redacted. Secrets, auth headers, API keys, env dumps,
and raw provider private payloads must not be exposed as stable event payloads.

### Bus

The first bus should be in-process and testable.

Required behavior:

- publish events in call order
- allow multiple subscribers
- isolate subscriber failures so one consumer cannot block the bus
- preserve a bounded recent event buffer for projections/debugging
- support dedupe by `dedupe_key`
- expose a small read API for the session activity projection

### Session Activity Projection

The first projection should track one row per provider/session pair:

```python
{
    "providerId": "codex",
    "workspaceId": "codex:onlineworker-combined",
    "workspacePath": "/Users/wxy/Projects/onlineworker-combined",
    "sessionId": "...",
    "title": "...",
    "status": "running|needs_attention|completed|failed|idle",
    "attentionReason": "",
    "lastUserMessage": "",
    "lastAssistantMessage": "",
    "lastFinalMessage": "",
    "lastEventKind": "message.assistant.final",
    "updatedAt": 1780000000.0,
}
```

TaskBoard should prefer this projection over ad hoc dashboard/session field
scraping.

### Notification Consumer

Notification routing should be represented as bus activity:

- `notification.requested`
- `notification.emitted`
- `notification.skipped`
- `notification.failed`

The notification summary logic should consume `message.assistant.final` and
`turn.completed` events as its input. This prevents notification summary from
becoming another private completion pipeline.

### Compatibility Strategy

Do not remove existing paths in the first slice.

Instead:

1. Publish bus events beside existing logic.
2. Add tests proving events match current behavior.
3. Make TaskBoard read projection data where available.
4. Keep fallback to existing session/dashboard fields until projection coverage
   is complete.
5. Move notification summary to the bus once final/turn events are covered.

## Implementation Tasks

1. Add core event contracts and bus. **Done**

   - Define canonical event kinds.
   - Add event id and dedupe helper functions.
   - Add `MessageEventBus` with bounded memory and failure-isolated
     subscribers.
   - Attach a bus instance to `AppState`.

2. Publish from provider session events. **Done**

   - Bridge `core.providers.session_events.SessionEvent` into bus events.
   - Map assistant delta/final, turn started/completed/failed,
     approval/question requested, and session created/title events.
   - Preserve provider-private raw payload handling for existing event logic.

3. Publish from user send paths. **Done**

   - After `prepare_user_message_text`, publish `message.user.accepted`.
   - Cover Telegram sends, owner bridge sends, provider session bridge sends,
     and App session sends.
   - Include source surface and redacted attachment metadata.

4. Add session activity projection. **Done**

   - Update projection on user accepted, assistant delta/final, turn status,
     approval/question, and failure events.
   - Expose projection through owner bridge or a Tauri command usable by
     TaskBoard.

5. Update TaskBoard. **Done**

   - Prefer session activity projection fields.
   - Show intuitive card information:
     provider, workspace, session id, title/current task, last message summary,
     status, attention reason, updated time.
   - Keep current pin/hide behavior.
   - TaskBoard must subscribe to the session activity stream instead of
     polling activity projection snapshots.
   - Running cards must keep the real session title as the card title and show
     latest activity/user content as the preview, clamped to exactly three
     lines without exposing a partial fourth line.
   - If a running activity has no assistant summary yet, show the last user
     message instead of leaving a large empty preview area.
   - Session Browser must expose a direct "follow session" action that writes
     the same TaskBoard pinned-session state, otherwise the "Pinned" lane has
     no usable entry point.
   - Pinned idle cards must also show the latest useful session message when
     available. The pinned lane is not only a status bucket; it must remain
     useful when the session is not currently running.

6. Fix Usage date range freshness. **Done**

   - The Usage tab default range must be based on the current local date when
     the tab refreshes or opens.
   - If the user has not manually changed the range, crossing midnight should
     move the default window forward. For example, on 2026-06-05 the default
     seven-day range should end at 2026-06-05, not remain at 2026-06-04.
   - If the user has manually applied a custom date range, refresh should
     preserve that custom range instead of overriding it.

7. Publish notification events. **Done**

   - Emit `notification.requested` before router delivery.
   - Emit `notification.emitted`, `notification.skipped`, or
     `notification.failed` after router result.
   - Preserve Phase 6 notification plugin delivery behavior.

8. Move notification summary input toward bus events. **Partially done**

   - Make final-reply summary consume the canonical final/turn event payload.
   - Keep current local/AI fallback behavior from Phase 8.

   14-01 now publishes canonical final/turn and notification activity events
   before existing notification delivery. The legacy summary generation remains
   behavior-compatible and now has a canonical bus input available for the next
   migration slice.

9. Add regression tests. **Done**

   Required coverage:

   - bus publishes events in order
   - duplicate dedupe keys do not update projections twice
   - subscriber exception does not prevent other subscribers
   - user sends from Telegram and owner bridge publish equivalent accepted events
   - provider final reply updates projection and notification input
   - approval requested event carries no global-topic fallback authority
   - TaskBoard model uses projection fields when present
   - notification emitted/skipped/failed is visible as bus activity

## Implementation Summary

14-01 added the first source-verified in-process message/event bus slice:

- `core/messages/` now contains the canonical `MessageEvent` contract, redacted
  public payload handling, bounded in-memory `MessageEventBus`, session-event
  bridge, publish helpers, and session activity projection.
- `AppState` owns a `message_bus` instance so runtime entry points can publish
  without creating surface-specific state.
- Telegram sends and App/session-tab owner-bridge sends publish comparable
  `message.user.submitted` and `message.user.accepted` events after the Phase 7
  user-message gateway resolves final text.
- Provider app-server/session events publish canonical `turn.*`,
  `message.assistant.*`, `approval.requested`, and `question.requested` events
  beside the existing Telegram/App delivery path.
- Telegram approval and question answers now publish `approval.answered` and
  `question.answered`.
- Notification delivery publishes `notification.requested`,
  `notification.emitted`, `notification.skipped`, and `notification.failed`
  events around the existing Phase 6 notification router.
- TaskBoard reads the session activity projection through the provider owner
  bridge and falls back to existing session/dashboard fields during migration.
- App navigation now exposes a TaskBoard tab, and TaskBoard cards can open the
  corresponding Session tab target.

## Risks

- Moving too much at once can regress Telegram final reply streaming. Keep the
  first slice additive.
- Event payloads can leak provider-private or secret data if raw payloads are
  copied blindly. Redaction is required at the bus boundary.
- If TaskBoard depends on Telegram-specific ids, it will recreate the coupling
  this phase is meant to remove.
- Notification summary can still duplicate final-reply logic unless its input is
  explicitly a canonical bus event.
