# 14-04 Plan: Fix TaskBoard Running Activity First Paint

**Status:** source and fast packaged verified
**Created:** 2026-06-05
**Phase:** 14. Unified Message Event Bus
**Trigger:** User UAT screenshot showed the Running lane count/card disappearing while the provider owner bridge still reported an active running session.

## Problem

The TaskBoard backend projection correctly reported a running Codex session, but
the frontend could still show an empty Running lane or a large blank card body.

Two UI-path issues caused the mismatch:

- The first `refresh` skipped `session_activities`, then waited for the slower
  provider session list and pinned preview hydration before the board could stop
  showing the loading/empty state.
- The live activity stream updated `sessionActivities`, but its snapshot and
  activity events did not clear `loading`, so the user could still see an empty
  lane while a running activity already existed.
- Running activity preview fallback preferred assistant/final content and the
  last user message, but did not use the activity title/session title when no
  message content was available.

## Goal

Make the Running lane reflect the bus-derived activity projection immediately.

The user-visible target behavior:

- If the provider owner bridge reports a running activity, the Running lane must
  not appear empty while the full session list is still loading.
- A running card should show useful preview text instead of a large blank body.
- Stream snapshot/activity events should be enough to bring the board out of
  loading state.
- Slower session metadata hydration should enrich cards later, not block the
  initial activity projection.

## Scope

In scope:

- Include activity projection in the initial TaskBoard refresh.
- Commit provider/dashboard/board/activity state before slower session list and
  pinned preview hydration completes.
- Clear `loading` on activity stream snapshot and activity events.
- Add preview fallback from last user message, activity title, or resolved title
  when assistant/final content is not yet available.
- Add regression coverage for the first-refresh activity path and title fallback
  preview.

Out of scope:

- App Session detail live rendering migration.
- Telegram rendering/topic migration.
- Persistent event replay/audit logs.
- Reworking provider session list loading beyond unblocking first paint.

## Implementation Summary

Completed on 2026-06-05:

- `TaskBoard.tsx` now calls `refresh({ includeActivities: true })` on initial
  mount.
- `TaskBoard.tsx` commits providers, dashboard state, TaskBoard pinned state,
  session activities, `nowMs`, and `loading=false` before fetching full provider
  session lists.
- TaskBoard activity stream snapshot and activity events now call
  `setLoading(false)` after applying the latest activity projection.
- `buildTaskBoardModel` now uses a running preview fallback from last user
  message, activity title, or resolved card title when no assistant/final
  content exists.
- Frontend tests cover the activity title preview fallback and assert that the
  TaskBoard first refresh includes activities.
