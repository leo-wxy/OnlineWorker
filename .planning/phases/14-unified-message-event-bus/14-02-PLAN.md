# 14-02 Plan: Harden TaskBoard Activity Stream Follow-ups

**Status:** source verified
**Created:** 2026-06-05
**Phase:** 14. Unified Message Event Bus
**Trigger:** source review of `6458990 feat(task-board): 接入消息总线会话看板`

## Problem

The current Phase 14 implementation has the right architecture direction:
TaskBoard now consumes bus-derived activity instead of re-deriving card state
from unrelated session/dashboard fields, and notification summary now has a bus
consumer path.

The follow-up review found three issues that should be fixed before treating the
TaskBoard activity stream as production-stable:

- A live activity event can be lost between the initial snapshot send and the
  later bus subscription.
- TaskBoard stream stop uses a global generation counter, so a stale cleanup can
  invalidate a newer stream during fast tab switches or React StrictMode-style
  lifecycle churn.
- Pinned idle session preview hydration can become expensive when many sessions
  are pinned.

## Goal

Make the TaskBoard activity stream lifecycle race-free and bounded while
preserving the Phase 14 event bus contract and current TaskBoard behavior.

## Scope

In scope:

- Provider owner bridge session activity stream startup ordering.
- TaskBoard Tauri stream start/stop lifecycle identity.
- TaskBoard pinned idle preview hydration bounds.
- Focused regression tests for event loss, stale cleanup, and hydration scale.

Out of scope:

- Replacing the in-process bus with an external broker.
- Rewriting provider adapter protocols.
- Changing TaskBoard lane/product semantics beyond the lifecycle hardening.
- Changing notification plugin settings UI.
- Packaged-app verification unless explicitly approved in the current
  conversation.

## Required Changes

1. Close the activity stream snapshot/subscribe race.

   Current risk:

   - `core/provider_owner_bridge.py` sends the initial `snapshot` payload before
     registering the bus subscriber.
   - If a bus event is published after the snapshot is captured but before the
     subscriber is registered, TaskBoard never receives that activity.
   - Because the new TaskBoard path relies on live activity instead of a periodic
     polling loop, the missed event can remain invisible for a long time.

   Required behavior:

   - Register the activity subscriber before the initial snapshot is sent, or
     introduce a watermark/replay handshake that guarantees no event is lost
     between snapshot creation and subscription activation.
   - Add a regression test that publishes an activity event during stream
     startup and proves TaskBoard receives either the updated snapshot or a
     subsequent activity event.

2. Make stream stop target a concrete stream instance.

   Current risk:

   - `mac-app/src-tauri/src/commands/task_board_state.rs` uses a global
     `TASK_BOARD_ACTIVITY_STREAM_GENERATION` counter for both start and stop.
   - `mac-app/src/pages/TaskBoard.tsx` cleanup calls the stop command without a
     stream id.
   - A stale cleanup from an older mount can increment the global generation and
     make a newer stream exit.

   Required behavior:

   - Give each started TaskBoard activity stream a concrete stream id.
   - Return that id from `start_task_board_activity_stream`.
   - Require `stop_task_board_activity_stream` to receive that stream id and
     ignore stale stop calls for older streams.
   - Add lifecycle tests covering quick start/stop/start ordering so an old
     cleanup cannot stop the latest stream.

3. Bound pinned idle preview hydration.

   Current risk:

   - TaskBoard reads latest session messages for pinned sessions during
     hydration.
   - If many historical sessions are pinned, opening TaskBoard can trigger a
     large number of session-message reads.

   Required behavior:

   - Hydrate only the pinned sessions that can become visible card candidates,
     or apply a small limit with stable ordering.
   - Keep the pinned lane useful for idle sessions while bounding initial load
     cost.
   - Add a focused test that many pinned sessions do not cause unbounded message
     reads.

## Implementation Summary

Completed on 2026-06-05:

- `core/provider_owner_bridge.py` now registers the activity stream subscriber
  before sending the initial snapshot. Events that arrive during startup are
  queued and flushed after the snapshot, so startup cannot lose a live activity.
- `start_task_board_activity_stream` now returns a concrete `streamId`.
  `stop_task_board_activity_stream` accepts that id and ignores stale stop calls
  for older streams.
- `TaskBoard.tsx` stores the active stream id, stops only that stream on cleanup,
  and handles the case where a component unmounts before `start` resolves.
- Pinned idle preview hydration is limited to the newest 12 visible pinned
  candidates, keeping the pinned lane useful without unbounded session-message
  reads.
- Regression coverage was added for startup event delivery, stale stream stop,
  stream-id wiring, and bounded pinned hydration.
