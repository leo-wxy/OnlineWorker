# 08-01: Add General AI Capability Layer and AI Configuration Tab

## Objective

Add a shared AI capability layer that can be reused by different OnlineWorker scenarios. The first scenario is notification completion summary, but the design must support future scenarios by changing scenario prompt/config rather than adding one-off summary logic.

## Product Direction

The app should add a top-level sidebar tab named `AI`. This tab is for configuring AI capability itself, separate from provider/agent setup and separate from notification channel setup.

The tab has two primary sections:

1. `Services`: connection-only configuration for upstream AI services.
2. `Scenarios`: per-use-case configuration for prompts, selected service, output schema, limits, and fallback behavior.

`Services` never owns prompt text. `Scenarios` owns prompt templates and decides which configured service to use for that scenario. The effective model is the model selected on that service card.

Recommended sidebar order:

```text
Dashboard -> Sessions -> Usage -> AI -> Commands -> Notifications -> Setup
```

AI sits before operational extension surfaces because it is a shared capability used by multiple features, while notifications and commands are concrete consumers.

## Configuration Model

Add a top-level `ai` configuration namespace.

Service connection configuration:

```yaml
ai:
  services:
    - id: openai_default
      name: OpenAI
      protocol: openai_compatible_chat
      base_url: https://api.openai.com/v1
      api_key: sk-...
      models:
        - gpt-5.4
      default_model: gpt-5.4
      timeout_seconds: 20
      enabled: true

    - id: claude_default
      name: Claude
      protocol: claude_messages
      endpoint: https://api.anthropic.com/v1/messages
      api_key: sk-ant-...
      models:
        - claude-sonnet-4-6
      default_model: claude-sonnet-4-6
      timeout_seconds: 20
      enabled: false
```

Scenario/function prompt configuration:

```yaml
ai:
  scenarios:
    notification_summary:
      enabled: true
      service_id: openai_default
      output_schema: notification_summary_v1
      fallback: local_notification_summary_rules
      limits:
        preview_title: 16
      prompt_template: |
        You summarize OnlineWorker task completion notifications.
        Return JSON with preview_title and summary.
        preview_title should identify the completed task.
        summary should explain the completed result.

        Current task:
        {{task_summary}}

        Final assistant message:
        {{final_message}}
```

Rules:

1. Product UI exposes fixed OpenAI and Claude service cards. Custom service creation is not part of the primary path for this phase.
2. `ai.services` contains only connection, auth, model, timeout, and health-check settings.
3. `ai.scenarios` contains prompt templates, output schemas, limits, selected service, enablement, and fallback policy.
4. A scenario must select one configured service. Multiple configured services are not called in priority order.
5. The scenario's effective model comes from the selected service's `default_model`.
6. A scenario with no valid selected service uses its fallback path.
7. Prompt config is scenario-scoped. Adding a future use case such as command explanation, issue triage, or release note generation means adding a scenario, not modifying service API settings.

Internal compatibility:

- The persisted config may keep `id` and `protocol` because runtime adapters need them.
- These fields are internal implementation details and must not be editable text fields in the user-facing AI tab.
- Legacy `api_key_env` configs may be read for compatibility, but the product UI should not ask users to configure environment variable names.

## Architecture

Add `core/ai/` as a provider-neutral product capability.

```text
consumer feature
  -> core.ai.scenarios.run_ai_scenario(scenario_id, variables)
  -> load scenario config
  -> resolve service config
  -> render prompt template
  -> call protocol adapter
  -> validate output schema
  -> return structured result or fallback signal
```

Core modules:

- `core/ai/contracts.py`: service config, scenario config, request/result dataclasses, schema validation result.
- `core/ai/config.py`: parsing and normalization for `ai.services` and `ai.scenarios`.
- `core/ai/templates.py`: small template renderer for scenario variables.
- `core/ai/client.py`: direct HTTP protocol adapters for OpenAI-compatible chat and Claude messages.
- `core/ai/scenarios.py`: scenario execution entrypoint, service resolution, timeout/error handling, fallback signaling.

This layer is not a provider plugin and does not call provider CLI runtimes. It is a direct reusable AI utility for OnlineWorker product features.

## UI Scope

Add `AI` as a first-class sidebar tab:

- Update `PRIMARY_APP_TABS` and `AppTab` to include `ai`.
- Add a sidebar icon and i18n labels for `AI`.
- Route `activeTab === "ai"` to `AiSettingsPanel`.
- Add source tests that assert the tab order and route are wired.

`AiSettingsPanel` layout:

- Match the Notification settings panel style: left navigation list, right detail editor, dense operational cards, and the enable switch in the detail header's top-right area.
- Top-level AI sections:
  - `Services`
  - `Scenarios`
- Services view:
  - Show fixed built-in service cards for OpenAI and Claude.
  - Do not make `protocol`, `service id`, or `api_key_env` editable UI fields.
  - Fields: API Key, request URL, supported models, selected model, timeout.
  - `Enabled` is in the service detail header top-right area.
  - Include a `Test connection` action that validates the current service draft.
  - Provide deletion/cleanup only for non-built-in or migrated custom services if they are shown; built-in OpenAI and Claude should remain available as product choices.
- Scenarios view:
  - List configured scenarios on the left.
  - Detail editor on the right.
  - Fields: enabled switch, service selector, output schema, limits, fallback policy, prompt template.
  - `Service` is a select/dropdown populated from configured services.
  - Do not ask users to type service ids.
  - Do not expose a scenario model field. The scenario uses the selected service's configured model.
  - The `notification_summary` scenario is present by default.

Design constraints:

- Keep the surface operational and dense.
- Do not place prompt fields inside the service connection editor.
- Do not place this under `Setup -> Agents`; this is shared AI capability configuration.
- Do not place notification-specific prompt controls in `Notifications`; notifications consume the configured AI scenario.
- Do not expose adapter/protocol details as product configuration controls.
- Do not expose environment variable names as required user input.
- Do not imply that all enabled services will be called. A scenario selects exactly one service.

## First Consumer: Notification Summary

Use `notification_summary` as the first scenario.

Input variables:

- `task_summary`: the current task or thread topic summary.
- `final_message`: the final assistant response.
- `agent_name`: display label such as Codex.
- `status`: notification status.
- `provider_id`: provider id when available.

Expected structured output:

```json
{
  "preview_title": "优化任务通知",
  "summary": "完成通知文案调整，并保留本地规则作为兜底。"
}
```

Behavior:

1. If the scenario is enabled and the selected service is valid, call that selected AI service.
2. If the call succeeds and output validates, use `preview_title` as the notification preview title and `summary` as the notification body summary.
3. If the scenario is disabled, times out, returns invalid output, or has no usable service, use the existing local notification summary rules.
4. The fallback path remains deterministic and testable.
5. Notification generation must not create a provider session, CLI session, or Telegram topic.
6. OpenAI and Claude may both be configured, but only the service selected by `notification_summary` is called for this scenario.

## Out of Scope

- Replacing agent/provider implementations.
- Creating a chat surface inside the AI tab.
- Running arbitrary user prompts from the AI tab.
- Adding vector memory, conversation history, or provider session persistence.
- Moving notification channel settings into the AI tab.
- Packaging, installing, or packaged-app verification in this planning slice.
