# 09-01 Plan: Add Provider-Backed Session Archive Action

## Objective

Add an Archive action to concrete sessions in the Session tab. The action must archive through the provider's real source operation, then update local OnlineWorker state after success. Failed or unsupported archive attempts must remain visible to the user and must not locally mark the session archived.

## Scope

- Add a Session tab right-click context menu entry for Archive.
- Add a visible row action for environments where right-click discoverability is poor.
- Add a frontend API wrapper for the Tauri archive command.
- Add a Tauri command that executes a real provider archive operation through:
  - the running provider owner bridge when available, or
  - a sidecar provider session bridge that still calls the provider's real archive hook.
- Persist `onlineworker_state.json` archived state only after the real provider archive succeeds.
- Merge locally persisted archived overlays back into provider session lists so a session archived at the real source remains visible in the Archived view even when the source no longer returns it in the active list.
- Return clear errors for providers that do not expose real archive support.
- Refresh the current provider list and clear selected archived sessions after success.
- Keep adjacent session operations aligned with provider capabilities: provider usage data is exposed through provider metadata and usage hooks, and `/token_usage` runs as a local bot command only in agent topics.

## Non-Goals

- No local-only archive or hide behavior.
- No Telegram topic archive/delete behavior from the Session tab.
- No packaged build or installed-app verification without explicit user approval.
- No new archive support for providers whose source API does not expose a real archive operation.

## Acceptance Criteria

1. Right-clicking a concrete Session tab row opens an Archive action, and the visible row action exposes the same command.
2. Clicking Archive invokes `archive_provider_session` with `providerId`, `sessionId`, `workspaceDir`, and the current session title when available.
3. The backend performs a real provider archive operation before touching local state.
4. If real archive fails, the UI shows an error and the row remains in its original list.
5. If real archive succeeds, local session state is marked archived, the provider list refreshes, and the archived overlay can populate the Archived tab even if the provider source omits archived rows from its active listing.
6. Claude or any provider without real archive support fails clearly instead of pretending success.
7. Focused Python, Rust, Node, and TypeScript verification passes.

## Adjacent Session Operation: Provider Usage

This phase also closes the provider usage operation work that sits next to Session archive in the user-visible session/agent operations surface:

- Provider manifests expose `capabilities.usage`.
- The Usage tab discovers visible, managed providers that advertise usage capability instead of hard-coding a fixed provider list.
- Built-in providers can parse local usage sources directly.
- Extension providers expose usage through provider usage hooks.
- `/token_usage` is registered as a local bot command and only runs in agent topics.
- `/token_usage` rejects thread-topic use with a clear scope message and does not forward the command into an active provider conversation.
- Unsupported providers return an explicit unsupported message instead of fake or empty usage data.
