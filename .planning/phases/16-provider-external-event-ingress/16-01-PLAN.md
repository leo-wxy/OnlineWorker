# 16-01 Plan: Define Plugin-Scoped External Event Ingress

**Status:** completed
**Created:** 2026-06-06
**Phase:** 16. Provider External Event Ingress

## Problem

Phase 14 built the message bus and TaskBoard projection, but external provider
sessions still depend on provider-specific ingress being present.

The bus can only project what it receives. If external Claude or
distribution-provided provider sessions do not produce normalized provider
events, TaskBoard will show stale, late, or missing cards.

The fix must be plugin-scoped:

```text
Claude plugin owns Claude external hook ingress.
Distribution-provided provider plugin owns its own external listener ingress.
Core/message bus only receives normalized events.
```

## Goal

Define and then implement provider-plugin-owned external event ingress for
Claude and distribution-provided provider plugins, using OpenCode-compatible
behavior only as a reference mechanism for compatible providers.

## Scope

In scope:

- Claude plugin references global Claude hook behavior.
- Claude plugin manages OnlineWorker hook entries in global
  `~/.claude/settings.json` with marker, dedupe, backup, and remove-only-own
  behavior.
- Claude plugin maps external hook lifecycle events to existing standard
  provider events.
- Distribution-provided provider plugins may reference OpenCode-compatible
  hook/listener behavior.
- Distribution-provided provider plugins own external listener/plugin
  registration and map events through their own event mapping.
- Plugin-level degraded status for disabled, failed, or callback-unreachable
  ingress.
- Source-level verification before packaged validation.

Out of scope:

- Implementing OpenCode as a provider.
- Adding OpenCode product support.
- Core hook/plugin lifecycle logic.
- Core provider-private payload parsing.
- TaskBoard provider-private discovery.
- App Session detail rendering migration.
- Telegram rendering migration.
- Approval/question command boundary.
- Large UI changes.

## Reference Anchors

Claude references:

- `~/.claude/settings.json`
- `OnlineWorker/plugins/providers/builtin/claude/python/hook_bridge.py`
- `OnlineWorker/plugins/providers/builtin/claude/python/adapter.py`

Distribution-provided provider references:

- local OpenCode plugin type definition with
  `event?: (input: { event }) => Promise<void>`
- provider-local event mapper
- provider-local adapter

Bus references:

- `OnlineWorker/core/messages/session_bridge.py`
- `OnlineWorker/core/messages/publishing.py`

These bus files are references only. They should not receive provider-specific
logic in Phase 16.

## Implementation Boundaries

Allowed implementation directories:

```text
OnlineWorker/plugins/providers/builtin/claude/
distribution package outside OnlineWorker/
```

Allowed planning docs:

```text
OnlineWorker/.planning/
```

Allowed tests:

```text
OnlineWorker/tests/test_claude_adapter.py
OnlineWorker/tests/test_claude_external_ingress.py
distribution-owned provider default tests
distribution-owned provider external ingress tests
distribution-owned provider event bus bridge tests
distribution-owned provider permission flow tests
```

Standard event tests may be touched only to assert normalized event delivery,
not to add provider-private behavior.

Forbidden implementation direction:

```text
core scans provider files
core manages hooks/plugins
core parses provider hook payloads
TaskBoard reads provider-private state
```

## Planned Work Items

1. Lock documentation and branch scope.

   - Add Phase 16 `CONTEXT.md`.
   - Add this `16-01-PLAN.md`.
   - Update `ROADMAP.md`.
   - Update `STATE.md`.
   - Explicitly record that OpenCode-compatible behavior is reference-only.
   - Explicitly record that implementation code is limited to provider plugin
     directories unless a later plan calls out a narrow exception.

2. Claude plugin external ingress design.

   - Define marker format for OnlineWorker Claude hook entries.
   - Define global `~/.claude/settings.json` merge behavior.
   - Preserve existing third-party and user hook entries.
   - Define disable/uninstall remove-only-own behavior.
   - Define degraded status when install or callback delivery fails.

3. Claude plugin event mapping.

   Minimum event targets:

   - `SessionStart` -> `session.created`
   - `UserPromptSubmit` -> `message.user.submitted` and `turn.started`
   - `Stop` / `SessionEnd` -> `message.assistant.final` and
     `turn.completed` using transcript/session lookup when needed
   - provider error -> `turn.failed`

   Claude token-level assistant delta streaming is not required for Phase 16.

4. Distribution-provided provider plugin external ingress design.

   - Use OpenCode-compatible plugin/listener behavior as reference.
   - Confirm actual provider listener/config surface before implementation.
   - Keep all provider config discovery and registration inside the
     distribution-owned plugin package.
   - Preserve existing user plugins/listeners.
   - Define marker/dedupe/remove-only-own behavior.
   - Define degraded status when listener registration or callback delivery
     fails.

5. Distribution-provided provider plugin event mapping.

   - Reuse the provider-local event mapper where possible.
   - Map external listener events to existing `app-server-event` envelopes.
   - Keep `/global/event` SSE as supplementary for OnlineWorker-owned serve,
     not as the primary external-session path.

   Minimum event targets:

   - session created/updated -> `session.created` / `session.title_updated`
   - user message -> `message.user.submitted` or `message.user.accepted`
   - step start -> `turn.started`
   - text part updated -> `message.assistant.delta`
   - text completed / step finish -> `message.assistant.final` and
     `turn.completed`
   - error -> `turn.failed`

6. Source-level verification plan.

   Automated checks should cover:

   - Claude hook settings merge preserves unrelated entries.
   - Claude hook settings install is idempotent.
   - Claude hook settings uninstall removes only OnlineWorker marker entries.
   - Claude lifecycle payloads map to standard events.
   - Provider listener/plugin config preserves unrelated entries.
   - Provider listener/plugin install is idempotent.
   - Provider listener/plugin uninstall removes only OnlineWorker marker
     entries.
   - Provider external events map to standard events through plugin-local code.
   - Standard bus projection still receives normalized events.

7. Live source validation plan.

   Source validation requires user-assisted live triggers:

   - Start source OnlineWorker bridge/runtime.
   - Enable Claude plugin external ingress in source mode.
   - User starts/sends in an external Claude session.
   - Verify bus receives `providerId=claude` session/user/turn/final events.
   - Enable distribution-provided provider plugin external ingress.
   - User starts/sends in the external provider.
   - Verify bus receives provider session/user/turn/delta/final events.

## Completion Criteria

This plan is complete when:

- Phase 16 docs are committed on a Phase 16 branch.
- The reference code and modification scope are explicit.
- The implementation phase can proceed without revisiting whether core or
  TaskBoard should own provider-specific ingress.

Phase 16 source-verified later when:

- Claude external session events enter the bus through Claude plugin ingress.
- Distribution-provided provider external session events enter the bus through
  plugin-owned ingress.
- No core/provider-specific hook or listener logic is added.
- TaskBoard updates from the existing bus projection.

Phase 16 complete later when:

- Source verification passes.
- Packaged build/install/restart validation passes after explicit approval.
- Installed app can safely install/update/remove its own hook/listener entries
  without damaging existing user tooling.

## Risks

- Claude hook payload fields for `UserPromptSubmit`, `Stop`, and `SessionEnd`
  need source-level sample verification before final mapper implementation.
- Compatible providers may not expose the exact same plugin configuration surface as
  OpenCode. In that case, Phase 16 should document the limitation and use only a
  plugin-owned fallback, not a core scan.
- Hook/listener config files are shared with other tools, so marker and
  remove-only-own behavior are mandatory.
