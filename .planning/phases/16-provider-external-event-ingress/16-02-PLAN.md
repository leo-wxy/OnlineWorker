# 16-02 Plan: Add Claude Plugin External Hook Ingress

**Status:** completed; automated source, installed-app live validation, and Computer Use UI validation passed
**Created:** 2026-06-06
**Phase:** 16. Provider External Event Ingress

## Problem

Claude external/manual sessions can run outside OnlineWorker's owned send path.
Phase 14's message bus can only project normalized provider events, so external
Claude sessions remain invisible or stale when the Claude provider plugin does
not listen to Claude lifecycle hooks and translate them into the existing event
chain.

The existing Claude plugin already has an OnlineWorker hook bridge for
approval/question behavior, and this slice now extends that bridge with a
global Claude hook merge plus external session lifecycle mapping.

## Goal

Add Claude external hook ingress inside the Claude provider plugin so externally
launched Claude sessions can publish normalized session/user/turn/final events
into the existing Phase 14 message bus.

## Design Requirements

1. Global settings merge is Claude-plugin-owned.

   - Merge OnlineWorker entries into global `~/.claude/settings.json`.
   - Do not replace the whole settings file.
   - Preserve existing third-party and user hook entries.
   - Support missing settings file by creating the minimal required structure.
   - Support malformed settings file with visible failed/degraded status instead
     of destructive rewrite.

2. OnlineWorker entries are marker-based.

   - Every OnlineWorker hook entry must carry a stable marker that can be found
     later.
   - Phase 16-02 will use the already-seen marker
     `onlineworker_claude_external_ingress_v1`.
   - Re-running install must dedupe/update OnlineWorker entries instead of
     appending duplicates.
   - Disable/uninstall removes only OnlineWorker marker entries.
   - Other hook entries must remain byte-for-byte or semantically preserved as
     much as structured JSON writing allows.

3. Hook events are lifecycle-focused.

   Minimum events:

   - `SessionStart`
   - `UserPromptSubmit`
   - `Stop`
   - `SessionEnd`

   Optional heartbeat/status events:

   - `PostToolUse`

   Timeout policy should follow the real installed shape:

   - blocking approval/question hooks keep `86400` second timeout
   - non-blocking lifecycle hooks use short timeouts such as `5` seconds

   Existing approval/question events remain supported:

   - `PreToolUse`
   - `PermissionRequest`
   - `Notification`

4. Hook payload mapping stays inside Claude plugin.

   Minimum mapping:

   - `SessionStart` -> `session.created`
   - `UserPromptSubmit` -> `message.user.submitted` and `turn.started`
   - `Stop` / `SessionEnd` -> `message.assistant.final` and
     `turn.completed`
   - provider error payload -> `turn.failed`

   `Stop` / `SessionEnd` may read Claude transcript/session data from inside
   Claude plugin code to find the last assistant message. Verified baseline hook
   input already includes `session_id`, `transcript_path`, `cwd`,
   `permission_mode`, and `hook_event_name`; `UserPromptSubmit` additionally
   carries `user_prompt`, while `Stop` carries `reason`. Core and TaskBoard must
   not perform that read.

5. Delta streaming is not required for Claude in 16-02.

   Claude hook payloads may not expose token-level assistant deltas. The minimum
   useful source-verified behavior is reliable user/turn/final events for
   external sessions. If a future hook payload exposes assistant delta content,
   it can be mapped by the Claude plugin later.

6. Failures are visible.

   Claude plugin should distinguish:

   - disabled
   - installed
   - install_failed
   - callback_unreachable
   - degraded_fallback

   Existing provider status surfaces can display this as text/details. Phase
   16-02 should avoid broad generic status model changes outside the plugin
   unless a later plan explicitly approves a narrow exception.

## Planned Work Items

1. Inventory current Claude hook payload paths.

   - Confirm current `--claude-hook-bridge` command behavior.
   - Confirm current socket relay behavior.
   - Confirm current approval/question hook handling remains intact.
   - Use the verified local hook docs as the baseline input contract:
     `session_id`, `transcript_path`, `cwd`, `permission_mode`,
     `hook_event_name`, `user_prompt`, and `reason`.
   - Confirm only the extra fields that matter for transcript lookup, workspace
     matching, and dedupe before final mapper behavior is locked.

2. Add global Claude settings helper inside Claude plugin.

   - Load `~/.claude/settings.json`.
   - Merge hook entries for required events.
   - Add marker metadata to OnlineWorker entries.
   - Deduplicate prior OnlineWorker entries.
   - Preserve unrelated hooks.
   - Provide uninstall/remove-only-own behavior.
   - Provide no-op behavior when external ingress is disabled.

3. Extend hook bridge event coverage.

   - Add OnlineWorker hook entries for lifecycle events.
   - Keep existing approval/question hook behavior.
   - Ensure command path works in source mode and packaged mode.
   - Ensure non-interactive lifecycle events return quickly and do not sit on
     the full approval timeout path unnecessarily.

4. Add Claude lifecycle hook mapper.

   - Extract session id, cwd/workspace, prompt text, transcript path, and error
     fields from hook payloads.
   - Resolve workspace id using existing Claude adapter workspace maps when
     available.
   - Emit existing `app-server-event` envelopes or normalized provider events
     that feed the current session event bridge.
   - Dedupe repeated lifecycle events by session/event/turn where possible.

5. Add final assistant extraction for stop/end events.

   - Use Claude plugin-local transcript/session readers.
   - Emit final assistant text when available.
   - Emit `turn.completed` even when final text is unavailable but the stop/end
     event is valid.
   - Emit degraded diagnostics when transcript lookup fails.

6. Add plugin-level diagnostics.

   - Record last install result.
   - Record last callback result.
   - Record last mapping error without leaking secrets.
   - Surface short status lines through existing Claude status builder if
     possible without changing shared status contracts.

7. Add focused source tests.

   Required coverage:

   - Merge creates lifecycle hook entries when settings file is missing.
   - Merge preserves unrelated third-party hook entries.
   - Merge is idempotent.
   - Uninstall removes only OnlineWorker marker entries.
   - Malformed settings returns failed/degraded result without destructive
     rewrite.
   - `SessionStart` payload maps to session created event.
   - `UserPromptSubmit` payload maps to user submitted and turn started events.
   - `Stop` / `SessionEnd` payload maps to assistant final and turn completed
     when transcript data is available.
   - Existing approval/question hook tests still pass.
