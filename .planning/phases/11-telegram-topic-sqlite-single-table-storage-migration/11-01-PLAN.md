# Plan 11-01: Migrate IM Route Bindings to One SQLite Table

## Goal

Make IM entry routing first-class durable storage so Telegram topic bindings, and
future IM bindings, cannot be erased by `onlineworker_state.json` runtime saves,
temporary state objects, cleanup, archive, streaming, or session bridge paths.

## Core Model

OnlineWorker owns the fixed internal route concepts:

- `agent`
- `workspace`
- `session`

IM platforms only provide external entries into those fixed targets. A Telegram
forum topic, Slack thread, Discord channel, Feishu chat, or custom WeChat bridge
entry is not the business object. It is only an entry bound to an OnlineWorker
route.

## Non-Negotiable Rules

- Use one SQLite table for route binding storage in the first implementation.
- Store routes in a generic IM model, not a Telegram-only topic table.
- Treat `(im_provider, im_account_id, im_space_id, im_entry_id)` as the primary
  identity of an IM entry.
- Migrate existing Telegram JSON topic ids once from `global_topic_ids`,
  workspace `topic_id`, and session/thread `topic_id`.
- After migration, SQLite is the truth source for IM entry lookup and routing.
- JSON `topic_id` fields are legacy migration input only. They must not be
  written back by `save_storage()` and must not be used as topic truth once
  SQLite is configured.
- Do not keep JSON and SQLite as competing truth sources.
- Do not use unknown IM entries as fallback routes.
- Do not physically delete route rows during cleanup/archive; use status fields.
- Keep route binding independent from IM display names. `display_name` is only
  metadata and must not drive routing.

## Table Shape

```sql
CREATE TABLE IF NOT EXISTS im_routes (
  im_provider TEXT NOT NULL,
  im_account_id TEXT NOT NULL DEFAULT 'default',
  im_space_id TEXT NOT NULL,
  im_entry_id TEXT NOT NULL,
  im_entry_kind TEXT NOT NULL DEFAULT 'unknown',

  route_scope TEXT NOT NULL,
  agent_provider TEXT,
  workspace_id TEXT,
  workspace_path TEXT,
  session_id TEXT,

  display_name TEXT,

  status TEXT NOT NULL DEFAULT 'active',
  source TEXT NOT NULL DEFAULT 'observed',

  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  last_seen_at INTEGER NOT NULL,

  PRIMARY KEY (
    im_provider,
    im_account_id,
    im_space_id,
    im_entry_id
  ),

  CHECK (route_scope IN ('agent', 'workspace', 'session', 'unknown')),
  CHECK (status IN ('active', 'archived', 'invalid', 'unknown')),
  CHECK (
    (
      route_scope = 'unknown'
      AND status = 'unknown'
      AND agent_provider IS NULL
      AND workspace_id IS NULL
      AND workspace_path IS NULL
      AND session_id IS NULL
    )
    OR (
      route_scope = 'agent'
      AND status IN ('active', 'archived', 'invalid')
      AND agent_provider IS NOT NULL
      AND workspace_id IS NULL
      AND session_id IS NULL
    )
    OR (
      route_scope = 'workspace'
      AND status IN ('active', 'archived', 'invalid')
      AND agent_provider IS NOT NULL
      AND workspace_id IS NOT NULL
      AND session_id IS NULL
    )
    OR (
      route_scope = 'session'
      AND status IN ('active', 'archived', 'invalid')
      AND agent_provider IS NOT NULL
      AND workspace_id IS NOT NULL
      AND session_id IS NOT NULL
    )
  )
);
```

Required indexes:

```sql
CREATE INDEX IF NOT EXISTS idx_im_routes_scope
ON im_routes(agent_provider, workspace_id, session_id);

CREATE INDEX IF NOT EXISTS idx_im_routes_seen
ON im_routes(last_seen_at);

CREATE UNIQUE INDEX IF NOT EXISTS uniq_active_im_agent_route
ON im_routes(im_provider, im_account_id, im_space_id, agent_provider)
WHERE route_scope = 'agent' AND status = 'active';

CREATE UNIQUE INDEX IF NOT EXISTS uniq_active_im_workspace_route
ON im_routes(im_provider, im_account_id, im_space_id, agent_provider, workspace_id)
WHERE route_scope = 'workspace' AND status = 'active';

CREATE UNIQUE INDEX IF NOT EXISTS uniq_active_im_session_route
ON im_routes(im_provider, im_account_id, im_space_id, agent_provider, workspace_id, session_id)
WHERE route_scope = 'session' AND status = 'active';
```

## Implementation Steps

1. Add an IM route store module that owns SQLite initialization, migration,
   reads, writes, WAL, and busy timeout.
2. Run JSON-to-SQLite migration during app/bot startup before Telegram handlers
   use message routing.
3. Stop serializing `global_topic_ids`, workspace `topic_id`, and thread
   `topic_id` to JSON. Legacy values loaded from JSON are used only by the
   one-time migration.
4. Replace topic lookup methods with SQLite-backed reads when the route store is
   configured:
   - `is_global_topic`
   - `get_tool_by_global_topic`
   - `find_workspace_by_topic_id`
   - `find_thread_by_topic_id`
   - workspace/session/global topic id lookup helpers used by notifications and
     event routing
5. Replace topic write points with SQLite writes where they assign topic ids:
   - global topic creation
   - workspace topic creation/rebuild
   - session topic creation/rebuild
   - received Telegram message observation
   - topic archive/invalid status changes where practical
6. Add `unknown` IM entry observation and make unknown entries fail closed with
   diagnostic output.
7. Add focused tests before packaging.

## Verification

- Migration test: existing JSON `global_topic_ids`, workspace `topic_id`, and
  session/thread `topic_id` produce `im_routes` rows.
- Routing test: deleting topic ids from in-memory JSON after migration does not
  break lookup through SQLite.
- Write isolation test: `save_storage()` cannot remove or change SQLite route
  rows.
- Unknown-entry test: unknown IM entry gets recorded and does not route to active
  workspace/session.
- Creation test: new agent/workspace/session entries are inserted into SQLite.
- JSON removal test: `save_storage()` no longer writes `global_topic_ids`,
  workspace `topic_id`, or thread `topic_id`.
- Existing slash/message/event routing tests continue to pass.

## Out of Scope

- Full migration of all runtime state to SQLite.
- MTProto/user-account topic enumeration.
- Automatic recovery of historical Telegram topics that are no longer present in
  JSON and have not sent a new update.
- Multi-table normalized workspace/session storage.
