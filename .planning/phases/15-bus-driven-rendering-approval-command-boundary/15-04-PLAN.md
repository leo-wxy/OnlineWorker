# 15-04 Plan: Define Approval And Question Command Boundary

**Status:** source verified
**Created:** 2026-06-08
**Phase:** 15. Bus-Driven Rendering And Approval Command Boundary
**Trigger:** Phase 15 decomposition after App/Telegram migration work was
separated from the higher-risk authority design decision.

## Problem

Phase 14 intentionally keeps approval/question bus events observational.
Phase 16 adds mirrored-only external CLI approval/question activity, which
reinforces that not every bus event implies OnlineWorker command authority.

Without an explicit boundary:

- event and command semantics can be conflated
- mirrored-only activity can be mistaken for owned actionable authority
- future approval/question handling can drift into unauthorized mutation paths

## Goal

Decide whether approval/question remains event-only or gains explicit command
objects/handlers, while preserving existing authority rules.

## Scope

In scope:

- Define event-vs-command semantics for approval/question flows.
- Preserve Codex app-server ownership for OnlineWorker-managed Codex sessions.
- Preserve native or mirror-only behavior for non-owned external sessions.
- Add focused authority regression tests.

Out of scope:

- Reworking provider ingress.
- Replacing Telegram route storage.
- Public plugin command API design.
- Packaged verification unless explicitly approved in the current conversation.

## Required Design Constraints

- Lifecycle events remain immutable and observational.
- Commands, if introduced, are separate objects/handlers.
- Command handlers must validate ownership before mutation.
- Mirrored-only approval/question activity remains non-actionable unless an
  explicit owned authority contract is later introduced.
- Non-owned Codex Desktop, VS Code, and ordinary CLI sessions remain native or
  mirror-only unless a controlled command path exists.

## Required Changes

1. Inventory current authority paths.

   - Codex app-server owned decisions
   - Telegram callback paths
   - provider owner bridge authority paths
   - mirrored-only observational paths

2. Choose the boundary.

   - Keep event-only if that remains the safest design, or
   - define explicit command objects/handlers with authority validation

3. Add focused regression tests.

   - unauthorized approval/question events cannot mutate state
   - mirrored-only events cannot become actionable commands
   - owned command paths, if added, only operate through validated owner/app-
     server paths

## Implementation Summary

- Approval lifecycle events remain observational on the bus; they do not create
  fallback command authority by themselves.
- Telegram approval handling is now fail-closed on explicit thread ownership:
  if the current session has no bound thread topic, the approval request is
  discarded instead of being retained as a local actionable decision path.
- Provider owner bridge approval replies now require a validated adapter
  `reply_server_request(...)` path. The old pending-decision fallback was
  removed so owner-bridge consumers cannot upgrade observational approval state
  into mutation authority without a real provider/app-server command route.
- The accepted controlled-host `codex_remote_proxy` path remains unchanged. It
  is still an explicit owned authority path from Phase 12 rather than a
  mirrored-only fallback.

## Follow-Up Discovered During Installed-App Investigation

Installed-app investigation after the Phase 15 source slices exposed a separate
distribution/config synchronization problem that is intentionally not counted as
part of 15-04 completion:

- Fresh installs currently start from `default-config.yaml`, while bundled
  provider and notification manifests, app-support `.env`, packaged
  `Info.plist` environment entries, and current process environment can all add
  more effective behavior.
- Rust/Tauri settings metadata and Python runtime config loading do not derive
  bundled-extension visibility from one shared effective-config model.
- Real installed-app evidence showed that bundled `codemaker` and `popo`
  resources were present under `/Applications/OnlineWorker.app/Contents/Resources`,
  but user-visible extension behavior still varied by machine because local
  config and overlay state changed what the app considered visible or enabled.
- Provider settings UI currently omits provider icon rendering, so even when
  metadata contains a resolved icon URL the App does not help explain what was
  discovered from bundled manifests versus local override state.

Required follow-up direction:

- treat bundled distribution extensions as first-class packaged resources rather
  than implicit overlay-only behavior
- unify Rust/Tauri and Python runtime effective-config semantics
- keep `config.yaml` focused on user overrides
- add clearer Settings diagnostics for source/default/current extension state
