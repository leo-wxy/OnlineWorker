# 15-04 Plan: Define Approval And Question Command Boundary

**Status:** planned
**Created:** 2026-06-08
**Phase:** 15. Bus-Driven Rendering And Approval Command Boundary
**Trigger:** Phase 15 decomposition after App/Telegram migration work was
separated from the higher-risk authority design decision.

## Problem

Phase 14 intentionally keeps approval/question bus events observational.
Phase 16 adds mirrored-only external CLI approval/question activity, which
reinforces that not every bus event implies OnlineWorker command authority.

Without an explicit boundary:

- event and command semantics can be conflated
- mirrored-only activity can be mistaken for owned actionable authority
- future approval/question handling can drift into unauthorized mutation paths

## Goal

Decide whether approval/question remains event-only or gains explicit command
objects/handlers, while preserving existing authority rules.

## Scope

In scope:

- Define event-vs-command semantics for approval/question flows.
- Preserve Codex app-server ownership for OnlineWorker-managed Codex sessions.
- Preserve native or mirror-only behavior for non-owned external sessions.
- Add focused authority regression tests.

Out of scope:

- Reworking provider ingress.
- Replacing Telegram route storage.
- Public plugin command API design.
- Packaged verification unless explicitly approved in the current conversation.

## Required Design Constraints

- Lifecycle events remain immutable and observational.
- Commands, if introduced, are separate objects/handlers.
- Command handlers must validate ownership before mutation.
- Mirrored-only approval/question activity remains non-actionable unless an
  explicit owned authority contract is later introduced.
- Non-owned Codex Desktop, VS Code, and ordinary CLI sessions remain native or
  mirror-only unless a controlled command path exists.

## Required Changes

1. Inventory current authority paths.

   - Codex app-server owned decisions
   - Telegram callback paths
   - provider owner bridge authority paths
   - mirrored-only observational paths

2. Choose the boundary.

   - Keep event-only if that remains the safest design, or
   - define explicit command objects/handlers with authority validation

3. Add focused regression tests.

   - unauthorized approval/question events cannot mutate state
   - mirrored-only events cannot become actionable commands
   - owned command paths, if added, only operate through validated owner/app-
     server paths
