# 15-03 Plan: Migrate Telegram Rendering To A Bus-Derived Edge Consumer

**Status:** source verified
**Created:** 2026-06-08
**Phase:** 15. Bus-Driven Rendering And Approval Command Boundary
**Trigger:** Phase 15 decomposition after confirming Telegram remains the
largest direct edge renderer outside TaskBoard.

## Problem

Telegram already receives lifecycle information that is published to the bus,
but its send/edit/topic/callback rendering path is still largely controlled by
direct edge logic.

That leaves several problems:

- Telegram rendering behavior is harder to explain as part of one canonical
  lifecycle.
- Route fail-closed, edit/send, parse mode, and callback behavior are preserved
  today, but they are not yet organized as a bus-derived edge consumer.
- Without a migration slice, Telegram-specific logic can continue to drift away
  from the shared event contract.

## Goal

Make Telegram rendering a bus-derived edge consumer while preserving Phase 11
route authority and Phase 12 approval ownership.

## Scope

In scope:

- Map current Telegram send/edit/final/streaming paths.
- Keep Telegram-specific presentation details at the edge.
- Preserve Phase 11 route storage and fail-closed behavior.
- Preserve Phase 12 approval ownership and callback semantics.
- Add focused regression coverage for Telegram rendering and routing.

Out of scope:

- App Session detail live model migration.
- Approval/question command-boundary implementation.
- Replacing Telegram-specific message ids or formatting with bus-level fields.
- Packaged verification unless explicitly approved in the current conversation.

## Current Implementation Anchors

- `bot/events.py`
- `bot/handlers/message.py`
- `bot/handlers/workspace.py`
- Any Telegram route lookup helpers used by the above flows

Current behaviors to map explicitly:

- direct `send_message`
- direct `edit_message_text`
- direct `edit_message_reply_markup`
- callback button ownership handling
- fail-closed route resolution from Phase 11

## Required Changes

1. Map current Telegram edge rendering behavior.

   - Identify streaming/final/edit/send paths.
   - Identify route lookup and invalid-route behavior.
   - Identify approval/question callback rendering and response paths.

2. Add a bus-derived Telegram edge consumer.

   - Keep chat/topic/message/parse/reply-markup details local to Telegram.
   - Drive rendering decisions from canonical bus events where feasible.
   - Prevent duplicate provider final/streaming renders.

3. Preserve route and ownership guardrails.

   - Missing/invalid routes must still fail closed.
   - Codex managed approval ownership must remain intact.
   - Mirrored-only approval/question activity must stay observational.

4. Add focused regression tests.

   - Telegram renders bus-derived streaming/final events.
   - Missing/invalid Telegram routes fail closed.
   - Mirrored-only approvals/questions do not create actionable controls.
   - Existing TaskBoard/notification consumers remain compatible.

## Implementation Summary

Completed on 2026-06-09:

- `core/messages/session_bridge.py` now makes the canonical message-event
  contract explicit for Telegram session rendering, including the existing
  product rule that non-Codex providers may emit a completed assistant message
  without an explicit `final_answer` phase while still intending a user-visible
  final reply. Semantic `turn_aborted` also now maps to canonical
  `turn.failed`, so abort rendering no longer depends on raw event branching.
- `bot/events.py` still preserves Telegram-specific send/edit/topic/message-id
  behavior at the edge, but its session-rendering decisions now read canonical
  `MessageEvent.kind` plus canonical payload fields (`text`, `delta`, `title`,
  `status`, `reason`) instead of re-deriving those decisions directly from
  provider-specific raw event shapes.
- Telegram session streaming/final/abort/topic-title behavior remains routed
  through the same fail-closed topic lookup, formatting helpers, and message id
  bookkeeping, but the decision boundary is now “canonical bus event → Telegram
  edge rendering”.
- Approval/question button ownership, callback handling, mirrored-only
  non-actionable behavior, and Phase 12 app-server authority remain on their
  existing path in this slice. This plan intentionally stops before the
  `15-04` command-boundary work.
- Focused regression coverage now locks both the canonical bridge behavior and
  the Telegram edge behavior for semantic final/abort sequences, including the
  non-Codex completed-without-phase final-reply rule.
