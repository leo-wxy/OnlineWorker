# 15-02 Plan: Migrate App Session Detail To A Bus-Derived Live Model

**Status:** source verified
**Created:** 2026-06-08
**Phase:** 15. Bus-Driven Rendering And Approval Command Boundary
**Trigger:** Phase 15 decomposition after Phase 16 closure confirmed that the
remaining migration debt lives in heavy consumers rather than provider ingress.

## Problem

App Session detail still depends on provider-specific read/send/stream paths.
The current desktop flow can list sessions, read turns, send messages, and
stream provider updates, but the live view model is not yet explained as a
canonical bus-derived consumer.

That leaves several problems:

- App Session detail still merges live updates through provider-specific stream
  semantics instead of one canonical event contract.
- History rows, assistant deltas, and final messages can only be reasoned about
  per provider path instead of one shared lifecycle model.
- Phase 14 proved the bus is the shared lifecycle source, but Session Browser
  detail still behaves like a separate live system.

## Goal

Make App Session detail rendering bus-derived while preserving provider history
reads for initial load and preserving current user-visible behavior.

## Scope

In scope:

- Map the current Session Browser detail read/send/stream entry points.
- Add an App Session bus-derived live model.
- Preserve provider history reads as initial/historical data only.
- Deduplicate history/live/final rows.
- Preserve loading, error, and scroll ergonomics.
- Add focused regression coverage for App Session live rendering.

Out of scope:

- Telegram send/edit/topic rendering migration.
- Approval/question command-boundary implementation.
- Provider protocol rewrites.
- Packaged verification unless explicitly approved in the current conversation.

- `start_provider_session_stream`
- provider-specific turn normalization and display modes

## Required Changes

1. Map current App Session live inputs.

   - Identify where initial history comes from.
   - Identify how live deltas/finals are merged today.
   - Identify provider-specific branches that only exist because the live model
     is not yet canonical.

2. Add a bus-derived live session model.

   - Convert canonical message/turn events into the Session Browser detail view
     model.
   - Keep provider history reads only as initial/historical hydration.
   - Ensure assistant delta/final ordering remains stable.
   - Ensure duplicate final rows are not rendered when history catches up.

3. Preserve behavior-critical UI ergonomics.

   - Scroll position should remain predictable during streaming.
   - Loading and error states should remain visible.
   - Markdown/plain display rules should remain intact.

4. Add focused regression tests.

   - App Session detail renders bus-derived assistant delta events.
   - App Session detail renders bus-derived final events.
   - History/live/final dedupe remains stable.
   - Existing TaskBoard/notification consumers remain compatible.

## Implementation Summary

Completed on 2026-06-09:

- `core/provider_owner_bridge.py` now exposes a dedicated
  `session_event_stream` request that subscribes to the internal message bus,
  filters by `provider_id` + `session_id` with optional `workspace_dir`, maps
  canonical bus events into Session Browser event payloads, and sends an
  initial `stream_ready` handshake before live updates.
- `mac-app/src-tauri/src/commands/provider_sessions.rs` now defines
  `ProviderSessionStreamEvent`, adds
  `start_provider_session_event_stream` / `stop_provider_session_event_stream`,
  and forwards owner-bridge session events over a dedicated Tauri
  `Channel<ProviderSessionStreamEvent>` without changing the legacy
  `start_provider_session_stream` Codex path.
- `mac-app/src-tauri/src/lib.rs` now registers the new provider session event
  stream commands.
- `mac-app/src/hooks/useProviderSessionEventStream.ts` adds a dedicated
  frontend hook for the new Tauri commands.
- `mac-app/src/components/session-browser/ClaudeChat.tsx` and
  `mac-app/src/components/session-browser/GenericProviderChat.tsx` now keep
  provider history reads as initial hydration only, ignore `stream_ready`,
  merge live bus-derived events through `applySessionStreamEvent`, preserve
  reply-watch cleanup, and degrade stream errors to non-destructive warnings
  when historical messages are already visible.
- `tests/test_provider_owner_bridge.py` now covers filtered
  `session_event_stream` delivery, attachment-aware user message mapping,
  assistant delta/final propagation, workspace filtering, and aborted-turn
  delivery after the `stream_ready` handshake.
- `mac-app/tests/providerSessionEventStream.test.mjs` locks the dedicated
  frontend/tauri wiring so future refactors do not silently collapse the new
  owner-bridge event stream back into the legacy stream path.
