# 15-01 Plan: Migrate Heavy Bus Consumers And Define Command Boundary

**Status:** planned
**Created:** 2026-06-05
**Phase:** 15. Bus-Driven Rendering And Approval Command Boundary

## Problem

Phase 14 establishes the internal unified message bus and migrates first
lightweight consumers such as TaskBoard and notification summary. It also
publishes App, Telegram, provider, approval, question, and notification
lifecycle events to the bus.

The heaviest rendering surfaces remain intentionally deferred:

- App Session detail live rendering still has provider/session stream behavior.
- Telegram still owns send/edit/topic rendering through existing edge logic.
- Approval/question bus events are observational only and do not execute
  answers.

Those choices keep Phase 14 manageable, but they leave migration debt that
should be handled after the event schema stabilizes.

## Goal

Move App Session detail live rendering and Telegram rendering toward bus-derived
consumers, then decide whether approval/question needs an explicit command
boundary separate from lifecycle events.

## Scope

In scope:

- App Session detail live updates from bus-derived events.
- Telegram bus-derived rendering/edit/topic edge consumer.
- Approval/question command-boundary design and implementation if approved by
  the plan.
- Regression coverage for App rendering, Telegram rendering, route authority,
  approval/question authority, and dedupe.

Out of scope:

- Public third-party plugin event API.
- Persistent canonical event audit/replay log.
- External broker.
- Provider adapter protocol rewrites.
- Notification plugin settings UI changes.
- Changing Phase 12 Codex app-server approval ownership semantics.

## Implementation Tasks

1. Map current App Session detail live rendering.

   - Identify current session stream/polling entry points.
   - Identify how history, assistant deltas, final messages, errors, and loading
     states are merged today.
   - Define which canonical bus events replace which live-rendering inputs.

2. Add an App Session bus-derived live model.

   - Convert canonical message/turn events into the session detail view model.
   - Preserve provider-specific history reads for initial load.
   - Deduplicate historical rows, live deltas, and final messages.
   - Preserve scroll and loading/error behavior.

3. Map current Telegram rendering edge behavior.

   - Identify send/edit/final/streaming paths.
   - Identify route lookup and fail-closed checks from Phase 11.
   - Identify approval/question button and callback rendering from Phase 12.
   - Define which canonical events can drive Telegram rendering without exposing
     Telegram-specific ids on the public bus contract.

4. Add a Telegram bus-derived edge consumer.

   - Keep Telegram-specific chat/topic/message/parse/reply-markup details at the
     Telegram edge.
   - Preserve Phase 11 route storage and fail-closed behavior.
   - Preserve Phase 12 approval ownership and callback semantics.
   - Ensure provider final/streaming events do not duplicate rendered messages.

5. Decide approval/question command boundary.

   - Keep lifecycle events immutable and observational.
   - If commands are introduced, define separate command objects/handlers.
   - Validate authorization before any provider/app-server state mutation.
   - Preserve Codex app-server request id ownership for OnlineWorker-managed
     Codex sessions.
   - Keep non-owned Codex Desktop, VS Code, and ordinary CLI sessions native or
     mirror-only unless a controlled command path exists.

6. Add focused regression tests.

   Required coverage:

   - App Session detail renders bus-derived assistant delta/final events.
   - App Session detail dedupes history/live/final rows.
   - Telegram renders bus-derived streaming/final events with existing route
     storage.
   - Missing/invalid Telegram routes fail closed.
   - Approval/question events cannot execute unauthorized commands.
   - Owned approval/question commands, if added, route only through validated
     owner/app-server paths.
   - Notification and TaskBoard consumers continue to receive canonical events.
