# 18-01 Plan: Unify App And Telegram New-Session Creation

**Status:** complete; source and packaged-app verified; real Telegram UAT waived by user
**Created:** 2026-07-05
**Phase:** 18. Provider Session New Flow

## Problem

The App Session tab now has a real provider-backed `New -> first message` flow,
but Telegram session-topic `/new` still needs to be explicitly locked down as
the same product operation.

In a concrete Telegram session topic, `/new <initial message>` should start a
separate real provider session under the current session's workspace. It must
not forward `/new` or the initial message into the current session.

## Goal

Make App and Telegram new-session behavior consistent:

- App `New` creates a real provider-backed session with the first message.
- Telegram session-topic `/new <initial message>` creates a real
  provider-backed session plus a new Telegram topic under the same workspace.
- No visible local draft session is treated as a real session.

## Scope

In scope:

- Characterize current Telegram `/new` behavior from a session topic.
- Add failing tests for the desired session-topic `/new` behavior.
- Reuse the existing Telegram new-thread handler if it already has the right
  workspace-resolution semantics.
- Patch slash routing or thread creation only where tests prove a gap.
- Keep provider-specific validation in provider thread hooks.
- Update user-facing docs only if command behavior changes materially.

Out of scope:

- Reworking the App Session `New` UI again.
- Creating a separate draft-session concept.
- Changing archive behavior.
- Changing approval/question ownership.
- Rewriting the message bus.
- Adding a new provider-specific Telegram command branch.

## Desired Telegram Flow

Input in an existing session topic:

```text
/new Explain this project
```

Expected behavior:

1. Slash router recognizes `/new` as an OnlineWorker command.
2. The command runs locally, not downstream.
3. The thread handler resolves the current session topic to its workspace.
4. Provider `start_thread(workspace_id)` creates a new provider session.
5. OnlineWorker creates a new Telegram forum topic.
6. OnlineWorker binds the new topic to the new session route.
7. The initial message goes through `prepare_user_message_text(...)`.
8. Provider send uses the new thread id.
9. The old topic receives only a confirmation/handoff message.
10. The new topic receives the thread control panel.

## Design Notes

### Prefer Existing Handler Shape

`bot/handlers/thread.py` already states the intended workspace priority:

```text
workspace Topic -> workspace
thread Topic -> thread's workspace
global Topic or legacy unknown -> active workspace
```

The first implementation step is therefore not to invent a new command path.
It is to test that this existing path behaves correctly for session-topic
`/new`.

### Command Rule

`bot/command_rules.py` currently defines `/new` as:

```text
executor=bot
scope=workspace
scope_policy=fallback
```

That can be acceptable if session-topic fallback means "run the workspace
command using this session's workspace." It is not acceptable if it causes
provider passthrough or current-session send.

Tests should decide whether this rule needs a narrow adjustment.

### Provider Validation

Codex validation currently rejects empty `/new`:

```text
codex 当前不能创建空 thread。
请使用 `/new <初始消息>`，这样源端 thread 才会 materialize...
```

Keep that behavior. Empty-session behavior should remain provider-owned, not
hard-coded into the slash router.

### Failure And Rollback

If provider session creation fails:

- do not create a persistent Telegram session route
- do not show a successful control panel
- report failure in the source topic

If Telegram topic creation fails after provider session creation:

- preserve existing rollback behavior where possible
- avoid leaving a visible fake/draft row
- report failure in the source topic

If initial message send fails after provider session and topic creation:

- use existing rollback behavior if that is the current handler contract
- otherwise document the state clearly and avoid claiming the new conversation
  is ready
