# 18-02 Plan: Converge Provider-Backed New-Session Core

**Status:** complete; source and packaged-app verified; real Telegram UAT waived by user
**Created:** 2026-07-05
**Phase:** 18. Provider Session New Flow

## Problem

Phase 18 behavior is now aligned at the product level:

- App `New` creates a real provider-backed session and sends the first message.
- Telegram `/new <initial message>` creates a real provider-backed session and
  routes the conversation to a new Telegram topic.

However, the implementation still duplicates the provider-backed "new session +
first message" core across two entry paths:

- `core/provider_owner_bridge.py` owner-bridge `start_session_message`
- `bot/handlers/thread.py` Telegram `/new`

This duplication is now architectural debt. The two paths can drift in:

- provider validation rules
- `start_thread` result handling
- `ThreadInfo` materialization
- first-message send semantics
- persistence and rollback behavior

## Goal

Extract a shared provider-backed new-session core service so App and Telegram
share one real-session creation contract while keeping their surface-specific
transport shells.

## Scope

In scope:

- Define a shared core helper/service under `core/` for real provider-backed
  new-session creation.
- Move common provider/session work out of owner bridge and Telegram `/new`.
- Keep provider-specific empty-session validation in provider hooks.
- Preserve current source-verified product behavior for both App and Telegram.
- Add focused regression tests around the shared core contract.

Out of scope:

- Reworking App Session UI or composer behavior.
- Reworking Telegram topic naming or control-panel UI.
- Rewriting owner bridge transport.
- Replacing Telegram route storage or rollback policy.
- Changing approval/question ownership.

## Convergence Rule

Do not converge App and Telegram by forcing them through one complete handler.

Converge only the provider-backed core:

1. resolve workspace/provider adapter
2. validate new-session request through provider hooks
3. call `start_thread`
4. reject local placeholder ids
5. materialize/update real `ThreadInfo`
6. send the first user message into the new real session
7. persist state and shape a unified result

Keep the outer transport shells separate:

- App shell:
  owner-bridge request/response, short-timeout `pending` return, session-tab
  optimistic UI match
- Telegram shell:
  source topic resolution, forum-topic creation, route binding, route-aware
  rollback, source-topic handoff message, control panel send

## Design Notes

### Shared Core Must Be Two-Stage

Telegram cannot simply call the App owner-bridge helper end to end, because TG
must bind the new Telegram topic before provider events for the new session can
arrive. That means the shared core should support a staged flow:

- Stage A: create and materialize the real provider session
- Stage B: send the initial user message into that real session

Telegram inserts topic create/bind between Stage A and Stage B.

App may still wrap the combined flow in a short timeout and return `pending`
while the background task finishes.

### Validation Should Become Shared

Telegram `/new` already uses `thread_hooks.validate_new_thread(...)`.
App `start_session_message` currently enforces only generic non-empty send
constraints.

The convergence slice should decide one shared validation entry so provider
rules stay in one place, with any source-specific differences made explicit
rather than accidental.

### Result Contract Should Stay Surface-Neutral

The shared core should return neutral session facts such as:

- provider id
- workspace id/path
- requested thread id
- effective thread id
- whether a new real thread was created
- whether thread id was remapped
- preview/session summary fields needed by callers

It should not return Telegram-specific topic ids or UI strings.

3. Refactor Telegram `/new` to use the same shared helper with a staged topic

4. Add focused tests for:
   - shared validation and invalid `start_thread` ids
   - App pending/slow-start behavior still preserved
   - Telegram source-topic reply and route binding still preserved
5. Re-run the existing Phase 18 regression suite plus any new shared-core
   tests.

## Acceptance Criteria

- App and Telegram share one provider-backed new-session core contract.
- Telegram still binds the new topic before the first message is sent.
- App still supports short-timeout `pending` return for slow provider startup.
- Provider-specific validation remains behind provider hooks.
- No visible local-only session is treated as a real provider session.
- Existing Phase 18 source-verified behavior remains green.
